#### 4.1.6 交叉验证引擎

```java
package tech.jxing.returnvision.service;

import lombok.extern.slf4j.Slf4j;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;

import java.util.ArrayList;
import java.util.Arrays;
import java.util.Collections;
import java.util.HashMap;
import java.util.List;
import java.util.Map;
import java.util.concurrent.CompletableFuture;

@Service
@Slf4j
public class OcrCrossValidatorService {

    @Autowired
    private OcrZhipuService ocrZhipuService;

    @Autowired
    private OcrAliyunService ocrAliyunService;

    /**
     * 双引擎并行OCR + 交叉验证
     *
     * 策略：
     *     1. 同时调用智谱OCR和阿里云OCR
     *     2. 逐字段比对两个引擎的结果
     *     3. 一致 -> 自动采用（高置信）
     *     4. 不一致 -> 标记差异字段 + 用阿里云valueProb辅助判断
     *     5. 完全失败 -> 转人工录入
     */
    @SuppressWarnings("unchecked")
    public Map<String, Object> dualEngineOcr(String imageUrl) {
        // 并行调用双引擎（均传入COS URL）
        CompletableFuture<Map<String, Object>> zhipuFuture = CompletableFuture.supplyAsync(() -> {
            try {
                return ocrZhipuService.ocrByZhipu(imageUrl);
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
        });
        CompletableFuture<Map<String, Object>> aliyunFuture = CompletableFuture.supplyAsync(() -> {
            try {
                return ocrAliyunService.ocrByAliyun(imageUrl);
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
        });

        // 分别获取结果，捕获异常以模拟 return_exceptions=True
        Map<String, Object> resultZhipu = null;
        Map<String, Object> resultAliyun = null;
        Exception zhipuEx = null;
        Exception aliyunEx = null;
        try {
            resultZhipu = zhipuFuture.get();
        } catch (Exception e) {
            zhipuEx = e;
        }
        try {
            resultAliyun = aliyunFuture.get();
        } catch (Exception e) {
            aliyunEx = e;
        }

        // 处理引擎失败情况
        if (zhipuEx != null && aliyunEx != null) {
            return Map.of("action", "manual", "reason", "双引擎均识别失败");
        }
        if (zhipuEx != null) {
            return Map.of("action", "accept", "data", resultAliyun, "source", "aliyun_only", "note", "智谱失败，仅阿里云结果");
        }
        if (aliyunEx != null) {
            return Map.of("action", "accept", "data", resultZhipu, "source", "zhipu_only", "note", "阿里云失败，仅智谱结果");
        }

        // 逐字段交叉比对
        List<String> fields = Arrays.asList("waybill_no", "rec_name", "rec_phone", "rec_address", "sender_name");
        Map<String, Object> matched = new HashMap<>();
        List<String> diffFields = new ArrayList<>();

        for (String field : fields) {
            String valZhipu = resultZhipu.get(field) == null ? "" : resultZhipu.get(field).toString().trim();
            String valAliyun = resultAliyun.get(field) == null ? "" : resultAliyun.get(field).toString().trim();

            if (!valZhipu.isEmpty() && !valAliyun.isEmpty() && valZhipu.equals(valAliyun)) {
                // 两引擎一致 -> 高置信采用
                Map<String, Object> m = new HashMap<>();
                m.put("value", valZhipu);
                m.put("status", "matched");
                matched.put(field, m);
            } else if (!valZhipu.isEmpty() && valAliyun.isEmpty()) {
                // 智谱有值，阿里云为空
                Map<String, Object> m = new HashMap<>();
                m.put("value", valZhipu);
                m.put("status", "zhipu_only");
                matched.put(field, m);
            } else if (!valAliyun.isEmpty() && valZhipu.isEmpty()) {
                // 阿里云有值，智谱为空
                Map<String, Object> m = new HashMap<>();
                m.put("value", valAliyun);
                m.put("status", "aliyun_only");
                matched.put(field, m);
            } else if (!valZhipu.isEmpty() && !valAliyun.isEmpty() && !valZhipu.equals(valAliyun)) {
                // 两引擎不一致 -> 标记差异
                Map<String, Object> confidenceMap = (Map<String, Object>) resultAliyun.getOrDefault("confidence", Collections.emptyMap());
                int prob = 0;
                Object probVal = confidenceMap.get(field + "_prob");
                if (probVal instanceof Number) {
                    prob = ((Number) probVal).intValue();
                }
                Map<String, Object> m = new HashMap<>();
                m.put("value", prob >= 80 ? valAliyun : valZhipu);
                m.put("alt_value", prob >= 80 ? valZhipu : valAliyun);
                m.put("status", "conflict");
                m.put("confidence", prob);
                matched.put(field, m);
                diffFields.add(field);
            }
        }

        // 判断整体策略
        Map<String, Object> waybillInfo = (Map<String, Object>) matched.getOrDefault("waybill_no", Collections.emptyMap());
        String waybillStatus = waybillInfo.getOrDefault("status", "").toString();
        if ("matched".equals(waybillStatus) && diffFields.isEmpty()) {
            // 关键字段一致且无差异 -> 自动采用
            Map<String, Object> data = new HashMap<>();
            for (Map.Entry<String, Object> entry : matched.entrySet()) {
                Map<String, Object> v = (Map<String, Object>) entry.getValue();
                data.put(entry.getKey(), v.get("value"));
            }
            Map<String, Object> result = new HashMap<>();
            result.put("action", "accept");
            result.put("data", data);
            result.put("source", "cross_validated");
            result.put("confidence", "high");
            return result;
        } else if (!diffFields.isEmpty()) {
            // 有差异 -> 标记供人工核对
            Map<String, Object> result = new HashMap<>();
            result.put("action", "review");
            result.put("data", matched);
            result.put("diff_fields", diffFields);
            result.put("confidence", "medium");
            return result;
        } else {
            // 关键字段不一致 -> 转人工
            Map<String, Object> result = new HashMap<>();
            result.put("action", "manual");
            result.put("reason", "关键字段不一致");
            result.put("data", matched);
            return result;
        }
    }
}

     * 业务规则：
     *     1. 运单号必填，且为纯数字或字母数字组合（8-30位）
     *     2. 电话号码格式校验（11位手机号或座机）
     *     3. 地址非空校验
     *     4. 交叉验证状态检查（如提供）
     */
    public ValidationResult validateRecord(Map<String, Object> record, Map<String, Object> crossValidation) {
        ValidationResult result = new ValidationResult();

        // 运单号必填
        Object waybillNo = record.get("waybill_no");
        if (waybillNo == null || waybillNo.toString().isEmpty()) {
            result.setValid(false);
            result.setMessage("运单号为空");
            return result;
        }
        // 运单号格式校验
        if (!WAYBILL_PATTERN.matcher(waybillNo.toString()).matches()) {
            result.setValid(false);
            result.setMessage("运单号格式异常：" + waybillNo);
            return result;
        }

        // 交叉验证状态检查
        if (crossValidation != null) {
            String action = crossValidation.getOrDefault("action", "").toString();
            if ("manual".equals(action)) {
                result.setValid(false);
                result.setMessage("双引擎识别不一致，需人工录入");
                return result;
            }
            Object diffFields = crossValidation.get("diff_fields");
            if (diffFields instanceof List) {
                @SuppressWarnings("unchecked")
                List<String> diffs = (List<String>) diffFields;
                if (!diffs.isEmpty()) {
                    result.setValid(true);
                    result.setMessage("校验通过（注意：" + String.join(", ", diffs) + " 字段存在差异，需人工核对）");
                    return result;
                }
            }
        }

        result.setValid(true);
        result.setMessage("校验通过");
        return result;
    }
}
```

     *     确认后的记录 -> 组装飞书格式 -> 调用batch_create -> 发送机器人通知 -> 返回写入结果

            fields.put("waybill_no", record.get("waybill_no"));

            fields.put("rec_phone", record.getOrDefault("rec_phone", ""));

            fields.put("status", "待处理");
            fields.put("photo_url", record.getOrDefault("image_url", ""));        // COS图片URL
            fields.put("return_reason", record.getOrDefault("return_reason", "")); // DeepSeek分析
            fields.put("return_category", record.getOrDefault("return_category", "")); // DeepSeek分类
            Map<String, Object> item = new HashMap<>();
            item.put("fields", fields);
            feishuRecords.add(item);
        }
        // 批量写入（单次最多500条）
        Object result = callFeishuBatchCreate(appToken, tableId, feishuRecords, token);

        }

        return result;
    }

#### 4.3.3 飞书机器人通知（⏸️ 暂不实现，v2.1 决策）

> **状态**：⏸️ 暂不实现（2026-07-18 决策）
>
> **原设计**：每条退货确认写入飞书后，发送卡片消息到飞书群（运单号+退货原因+分类+查看照片按钮）。
>
> **暂不实现理由**：
> 1. **业务价值低**：飞书多维表格是真正的数据出口，群通知夹在中间两头不讨好。客服自己录入的不需要通知，主管/售后/管理层要的是汇总而非逐条。
> 2. **群消息轰炸**：日均 100 单 = 每天 100 条卡片，峰值上千条，会淹没群内真正重要的沟通。
> 3. **F04 报表是更优解**：docs/10 第 2 期的多维度退货报表（原因 TOP10 / 快递占比 / 品类趋势）才是"让数据被看见"的正确姿势。
> 4. **需求可被替代**：若有人需要"被动接收行变更"，飞书多维表格自带行变更通知功能，无需我们开发。
>
> **`feishu.bot-webhook` 配置项当前仅用于 F12 异常告警**（见第 4.5 节），不用于业务通知。
>
> **何时重新评估**：出现明确的"特定角色需要每条退货都进群"的业务诉求时再讨论。

### 4.5 异常监控告警方案（F12，v2.1 新增）

> 关联：[docs/10-产品演进与功能可行性方案.md](./10-产品演进与功能可行性方案.md) 第 7.4 节 F12
> 范围：第 1a 期快速止血项，复用现有飞书机器人 Webhook，不动表结构、不改 API 契约
> 状态：✅ 已落地（2026-07-18）

#### 4.5.1 设计目标

- **感知**：让线上"沉默错误"（OCR 双失败、飞书写入连续失败、COS 上传失败、应用启动失败）可被运维即时感知
- **去重**：同类告警 5 分钟内只发一次，避免告警风暴
- **分级**：按严重程度分 WARN / ERROR / CRITICAL 三级，飞书卡片用不同颜色区分
- **不阻塞业务**：告警发送异步执行，失败也不影响主流程
- **优雅降级**：Webhook 未配置时静默跳过（与 `FeishuService` / `CosClientService` 已有降级模式一致）

#### 4.5.2 模块结构

```
tech.jxing.returnvision.common.alert
├── AlertLevel.java      # 告警级别枚举（WARN/ERROR/CRITICAL）
└── AlertService.java    # 告警服务（飞书机器人 Webhook + 去重 + 异步发送）
```

> 放在 `common.alert` 而非 `feishu` 包：告警是横切关注点，未来可能扩展邮件/钉钉等通道，不应耦合到飞书包。

#### 4.5.3 AlertLevel 级别定义

| 级别 | 飞书卡片颜色 | 触发场景 | 备注 |
|------|------------|---------|------|
| WARN | 黄色 | 单次 OCR 双失败、单次 COS 上传失败 | 可恢复，提示性 |
| ERROR | 橙色 | 飞书写入连续失败 ≥ 3 次、GlobalExceptionHandler 捕获的系统异常 | 需运维介入 |
| CRITICAL | 红色 | 应用启动失败 | 立即介入，服务可能不可用 |

#### 4.5.4 AlertService 核心方法

```java

    /**
     * 发送告警（自动去重 + 异步发送）
     *
     * @param level     告警级别
     * @param alertType 告警类型 key（用于去重，如 "ocr_dual_fail"）
     * @param message   告警消息（不含敏感信息）
     * @param context   上下文（如 cosUrl / record_id / exception 类名）
     */
    public void notify(AlertLevel level, String alertType, String message, Map<String, Object> context);

    /** 飞书写入失败计数（达到阈值时触发告警） */

    public void resetFeishuFailureCount();

#### 4.5.5 告警去重设计

- **存储**：`ConcurrentHashMap<String, Long>` -- key 为 `alertType`，value 为上次发送时间戳
- **判定**：`now - lastSent < dedupWindowMs` 时跳过发送
- **窗口**：默认 5 分钟（`alert.dedup-window-ms: 300000`）
- **过期清理**：每次 notify 顺带清理超过 2 倍窗口的过期 key，避免内存泄漏
- **特殊**：飞书写入失败用独立计数器 `AtomicInteger`，不进 dedup map（避免单次失败被去重掩盖连续失败）

#### 4.5.6 埋点位置（4 处）

| # | 位置 | 级别 | alertType | 上下文 |
|---|------|------|-----------|--------|
| 1 | `UploadController.uploadSse/upload/doUploadInternal` 中 OCR 双失败分支（`BizException(2001)` 抛出前） | WARN | `ocr_dual_fail` | cosUrl |
| 2 | `CosClientService.uploadToCos` 抛 `CosError` 前 | ERROR | `cos_upload_fail` | filename |
| 3 | `UploadController.confirmSingle` 调用 `feishuService.writeRecord` 抛 `FeishuApiError` 时 | ERROR | `feishu_write_fail` | record_id（达到 3 次阈值才发） |
| 4 | `ReturnVisionApplication` 监听 `ApplicationFailedEvent` | CRITICAL | `app_start_fail` | 异常类名 + message |

> GlobalExceptionHandler 捕获的"系统异常"（`Exception` 兜底分支）也接入 `AlertService.notify(ERROR, "system_error", ...)`，让未预期的运行时异常也能被感知。

            .eq(ReturnRecord::getStatus, "synced"));
    if (syncedCount > 0) {
        throw new DuplicateWaybillError("该运单已写入飞书，请勿重复上传：" + waybillNo);
    }

    Long confirmedCount = recordMapper.selectCount(
        new LambdaQueryWrapper<ReturnRecord>()
            .eq(ReturnRecord::getWaybillNo, waybillNo)
            .eq(ReturnRecord::getStatus, "confirmed"));
    if (confirmedCount > 0) {
        warnings.add("该运单已有待写飞书记录，请核对：" + waybillNo);
    }
}
```

#### 4.6.5 埋点位置（3 处）

| # | 位置 | 说明 |
|---|------|------|
| 1 | `UploadController.upload`（同步接口） | OCR 双失败校验之后、`recordMapper.insert` 之前 |
| 2 | `UploadController.uploadSse`（SSE 接口） | 同上，SSE 流式接口 |
| 3 | `UploadController.doUploadInternal`（批量复用） | 同上，批量上传内部方法 |

> 去重逻辑封装为 `checkDuplicate(waybillNo, warnings)` 私有方法，3 处复用。

#### 4.6.6 处理力度

| 场景 | 处理 | 客服感知 |
|------|------|---------|
| waybill_no 已有 synced 记录 | 抛 `DuplicateWaybillError(2002)`，阻断 | 前端收到错误提示"该运单已写入飞书，请勿重复上传" |
| waybill_no 已有 confirmed 记录 | 返回 warning，不阻断 | 前端收到警告"该运单已有待写飞书记录，请核对" |

| 正常 | 继续 insert | 无感知 |

#### 4.7.5 SecurityConfig 核心配置

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity  // 启用 @PreAuthorize
public class SecurityConfig {

    // 路径权限规则：
    // - /api/auth/** 公开（登录/刷新/飞书回调）
    // - /api/admin/** 需 ADMIN 角色
    // - /api/dashboard/** 需 SUPERVISOR 或 ADMIN 角色
    // - /api/** 其他接口需登录
    // - 静态资源 / / 公开
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())  // 前后端分离，用 JWT 不需 CSRF
            .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .requestMatchers("/api/dashboard/**").hasAnyRole("SUPERVISOR", "ADMIN")
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

#### 4.7.6 JWT 设计

**access token payload**：
```json
{
  "sub": "1",                    // user_id
  "username": "admin",
  "roles": ["ADMIN"],
  "type": "access",
  "exp": 1721300000,             // 2h 后
  "iat": 1721292800
}
```

**refresh token**：
- 不用 JWT，用随机字符串（UUID + SHA-256 哈希存库）
- 存库字段：`token_hash`（SHA-256 哈希）+ `user_id` + `expires_at`
- 刷新时：解析 token -> SHA-256 哈希 -> 查库比对 -> 签发新 access token

#### 4.7.7 OAuth 首次登录策略

> ⚠️ **首次部署必读**（2026-07-20 补）
>
> 飞书 OAuth 登录的前提：**用户的飞书 user_id 必须先在 `sys_user` 表中绑定对应记录**。否则点飞书登录后端会抛 `1004 飞书账号未绑定`，前端会停在登录页（已修复前端会显示错误信息，commit 8279cc6）。
>
> **绑定步骤**（部署时管理员必做）：
>   1. 先用账号密码登录系统（用 `admin` 或任意已存在用户）
>   2. 进入"个人中心"，在"飞书 user_id"字段填入自己的飞书 user_id
>   3. 如果不知道飞书 user_id，先点一次飞书登录走完授权（会报未绑定）
>   4. 这时后端日志会有 `[鉴权] 飞书账号未绑定：feishu_user_id=xxx`，把这个 user_id 复制出来
>   5. 回到个人中心填入这个 user_id 保存，再点飞书登录即可成功
>
> **批量绑定**（部署时管理员 SQL 一次性做）：
> ```sql
> -- 先让每个用户点一次飞书登录拿到 user_id（看后端日志）
> -- 然后执行：
> UPDATE sys_user SET feishu_user_id='<飞书user_id>' WHERE username='<系统用户名>';
> ```

- 用户用飞书 OAuth 登录时，后端用 feishu_user_id 查 sys_user
- 找不到 -> 返回 1004 错误"飞书账号未绑定，请联系管理员绑定"
- 管理员在用户管理界面创建用户后，可填入 feishu_user_id 完成绑定
- **不自动创建账号**：企业级"先审批后访问"原则

#### 4.7.8 验收要点

- 未登录访问 `/api/upload` 返回 401
- 客服（STAFF）访问 `/api/admin/**` 返回 403
- 客服访问 `/api/dashboard/**` 返回 403
- 账号密码登录成功返回 access_token + refresh_token
- access token 过期后用 refresh token 换新 access token
- 登出后 refresh token 失效，不能再用它换 access token
- 管理员禁用账号后，该账号 refresh token 立即失效
- 飞书 OAuth 登录，已绑定账号能登录，未绑定返回 1004
- 初始密码 admin123 登录后 must_change_password=true，强制改密
- SSE 接口能正确鉴权（fetch 手动注入 Authorization Header）

### 4.9 数据保留期与导出管控方案（F02，v2.1 新增）

> 关联：[docs/10-产品演进与功能可行性方案.md](./10-产品演进与功能可行性方案.md) 第 7.2 节 F02
> 范围：第 1b 期安全合规，业务数据生命周期管理 + 导出管控
> 状态：✅ 已落地（2026-07-19）
> 依赖文档：[docs/05 第 4.5.10 节](./05-数据库设计.md)（保留期配置）

#### 4.9.1 设计目标（回归 B 端业务本质）

- **不做界面脱敏**：内部系统靠权限（F01）+ 审计（F03）已足够，脱敏破坏业务价值
- **保留期管理**：业务数据按生命周期自动清理，避免无限增长
- **导出管控**：防止内部人批量拖走数据（即时导出 + 强审计 + 水印 + 限流 + 告警）
- **超期预警**：业务流程闭环，防止退货遗忘处理

#### 4.9.2 模块结构

```
tech.jxing.returnvision
├── retention/                          # 保留期管理（新增包）
│   ├── RetentionScheduler.java         # 定时任务（每天凌晨 2 点）
│   └── RetentionProperties.java        # 保留期配置（@ConfigurationProperties）
├── export/                             # 导出管控（新增包）
│   ├── ExportService.java              # 导出业务逻辑（含水印）
│   ├── ExportRateLimiter.java          # 限流（单次1000条+每日5次）
│   └── ExcelWatermarkUtil.java         # Excel 水印工具
└── controller/
    └── ExportController.java           # 导出接口
```

#### 4.9.3 保留期管理（定时任务）

Spring `@Scheduled(cron = "0 0 2 * * ?")` 每天凌晨 2 点执行：

| 数据类型 | 保留期 | 到期处理 | 配置项 |
|---------|:---:|---------|------|
| 退货记录（synced） | 90 天 | 物理删除（DELETE） | `retention.return-record.days=90` |
| 退货记录（pending） | 30 天 | 标黄提醒（不删除） | `retention.pending-remind.days=30` |
| 审计日志 operation_log | 180 天 | 物理删除 | `retention.audit-log.days=180` |
| OCR 日志 ocr_log | 30 天 | 物理删除 | `retention.ocr-log.days=30` |

**物理删除理由**：
- 90 天前的退货业务价值极低（业务闭环已完成）
- 释放存储，避免 return_records 表无限增长影响查询性能
- operation_log 保留了"谁导出过什么"的溯源记录，即使原数据删除也能追溯

**定时任务实现要点**：
- 分批删除（每批 500 条），避免大事务锁表
- 每批之间 sleep 100ms，降低 DB 压力
- 删除前先记一条 operation_log（action=RETENTION_CLEANUP）
- 异常时记错误日志，不影响下次执行

#### 4.9.4 超期预警

| 场景 | 阈值 | 动作 |
|------|------|------|
| 待确认退货未处理 | 24 小时 | 列表标黄 + Dashboard 显示数量 |
| 待确认退货积压 | > 50 条 | 飞书群通知主管（复用 AlertService） |
| 同步失败孤儿数据 | 7 天 | 管理员告警 |

**Dashboard 接口扩展**：`/api/dashboard/stats` 增加 `overdue_count` 字段（待确认 > 24h 的数量）

#### 4.9.5 导出管控（4 重防护）

**本期方案：即时导出 + 强审计**（不做审批流，审批流留 2 期）

| 防护层 | 实现 |
|-------|------|
| 1. 强制审计 | action=EXPORT_RECORDS 记录到 operation_log（复用 F03 的 @AuditLog） |
| 2. 水印 | Excel 每页页眉含"导出人:xxx 时间:xxx IP:xxx"，溯源到人 |
| 3. 限流 | 单次最多 1000 条；每人每日最多 5 次（内存计数 + 定时重置） |
| 4. 告警 | 导出后飞书群通知主管（"xxx 导出了 N 条退货数据"） |

**导出接口设计**：
```
POST /api/records/export
请求：{ status, start_date, end_date }  // 筛选条件
响应：Excel 文件流（application/vnd.ms-excel）
权限：STAFF/SUPERVISOR/ADMIN 都可导出（按角色过滤数据范围，复用 F03 的客服范围细化）
```

#### 4.9.6 导出限流规则

- **单次限制**：最多 1000 条（防止一次性拖走全表）
- **每日限制**：每人每日最多 5 次（防止频繁导出）
- **实现**：`ExportRateLimiter` 用 ConcurrentHashMap 存"用户ID+日期 -> 次数"，每天 0 点重置
- **超限响应**：返回 1011 错误码"导出次数超限"

#### 4.9.7 Excel 水印设计

每页页眉含：
- 导出人用户名 + 显示名
- 导出时间（精确到秒）
- 导出人 IP
- 数据条数

**为什么用页眉不用背景水印**：Apache POI 的背景水印实现复杂且易被移除，页眉简单且每页都有，溯源效果一样。

#### 4.9.8 验收要点

- 定时任务每天凌晨 2 点执行，清理超期数据
- 退货记录 90 天后物理删除
- 审计日志 180 天后物理删除
- OCR 日志 30 天后物理删除
- 待确认退货 > 24h 在列表标黄
- 待确认退货 > 50 条主管收到飞书通知
- 导出 Excel 含水印（导出人+时间+IP）
- 单次导出超 1000 条返回 1011
- 每人每日导出超 5 次返回 1011
- 每次导出都有审计记录（action=EXPORT_RECORDS）
- 导出后主管收到飞书群通知

### 4.10 OCR 准确率仪表盘方案（F05，v2.1 新增）

#### 4.10.1 设计目标

- **质量监控仪表盘**：可视化展示 OCR 识别质量趋势
- **故障定位**：某批退货识别异常时，能查是哪个引擎、哪个字段出问题
- **优化依据**：知道哪个字段常错，指导 prompt 或引擎调优

#### 4.10.2 模块结构

│   └── OcrCrossValidatorService.java  # 修改：dualEngineOcr 末尾加埋点

#### 4.10.3 字段级置信度埋点（核心）

**埋点位置**：`OcrCrossValidatorService.dualEngineOcr()` 方法末尾，在 return 前

**埋点逻辑**：每次 dualEngineOcr 调用写 2 条 ocr_log（智谱 1 条 + 阿里云 1 条）

| 字段 | 智谱 OCR | 阿里云 OCR |
|------|---------|-----------|
| engine | `zhipu_ocr` | `aliyun_waybill` |
| record_id | 本次识别关联的 record_id（若有） | 同左 |
| duration_ms | 智谱调用耗时 | 阿里云调用耗时 |
| success | zhipuSuccess | aliyunSuccess |
| confidence | 表级（成功=1.0/失败=0.0） | 阿里云返回的整体置信度 |
| field_confidence | `{}`（智谱无字段级） | `{"waybill_no":0.95,"rec_name":0.88,...}`（阿里云 valueProb） |
| error_msg | 失败时的异常信息 | 失败时的异常信息 |

**实现要点**：
- 用 `OcrLogWriter` 工具类封装组装+插入，避免污染 OcrCrossValidatorService 主流程
- 埋点失败不影响主流程（try-catch 吞异常，只记 warn 日志）
- 在 dualEngineOcr 的 3 个 return 点（双失败/单失败/正常）都埋点
