# 10 - 产品演进与功能可行性方案

## 一、文档目的与定位

本文档以产品经理视角，对「退运智录（ReturnVision）」当前能力做一次系统盘点，识别产品演进路径上的**结构性缺口**与**高价值增强点**，输出一份可被工程团队评估、可被业务方拍板的可行性方案。

> 重要前提：用户明确表示"允许项目保持不变"。因此本文档**不要求立即修改现有代码**，所有建议均为**候选演进项**，按优先级排序供后续选择性落地。

## 二、产品现状盘点

### 2.1 已具备的能力（值得肯定）

| 维度 | 现状 | 评价 |

| 核心闭环 | 拍照→COS→双引擎OCR→交叉验证→DeepSeek分析→人工确认→飞书写入 | ✅ 主干完整，业务闭环成立 |

### 2.2 产品定位再认识

当前定位：**"退货面单录入工具"** —— 帮客服把面单信息从照片搬进飞书表格。

潜在定位：**"退货运营平台"** —— 不仅能录入，还能洞察退货原因、指导供应链与售后改进。

> 录入是入口，洞察是价值。当前止步于"录入"，没有把数据价值反哺回业务，是最大的产品演进空间。

## 三、用户与场景分析

### 3.1 关键用户角色

| 角色 | 当前覆盖 | 核心诉求 |
|------|---------|---------|
| 一线客服 | ✅ | 快速录入、减少手敲、批量处理 |
| 客服主管 | ⚠️ 部分 | 处理效率监控、异常介入、团队绩效 |
| 售后/质量负责人 | ❌ | 退货原因趋势、品类问题定位、供应链反馈 |

| IT/运维 | ⚠️ 部分 | 系统健康、密钥轮换、故障告警 |

### 3.2 真实使用场景的隐含要求

- 客服在**仓库/快递站点**用手机操作 → 移动端体验、弱网下断点续传
- 客服换班 → 待确认记录需要"我的待办"边界，**避免互相覆盖**
- 退货高峰（如双11后） → 单日数百乃至上千单，**批量与队列能力**是刚需
- 飞书多维表格被多方查阅 → 飞书写入是**最终数据出口**，对一致性要求高

## 四、缺口诊断（产品视角的结构性问题）

按"价值 × 风险"二维评估，识别出 **6 类结构性缺口**：

| # | 缺口 | 风险等级 | 价值空间 | 说明 |
|---|------|---------|---------|------|
| G1 | **无登录与权限体系** | 🔴 高 | 高 | 任何拿到域名者即可使用，退货数据含客户姓名/电话，存在数据安全与合规风险 |
| G2 | **个人隐私数据明文流转** | 🔴 高 | 中 | rec_phone/sender_phone 明文存 MySQL 且参与 LLM 分析，无脱敏、无保留期 |
| G3 | **无统计报表/数据洞察** | 🟡 中 | 极高 | 仪表盘仅 KPI+7日趋势，无法回答"哪家快递退货多？哪类商品有问题？" |
| G4 | **无人工反馈闭环** | 🟡 中 | 高 | 客服修正后的结果未回流，LLM 分类准确与否无法度量，模型无持续改进依据 |
| G5 | **运单号无去重/格式校验** | 🟡 中 | 中 | 同一运单重复上传会重复入飞书，造成数据脏污 |
| G6 | **无异常监控告警** | 🟡 中 | 中 | OCR 双失败、飞书写入连续失败、COS 失败都"沉默"，运维后知后觉 |

### 第 1 期：安全底座（建议优先，2~4 周）
- **F01 用户登录与权限体系** —— 进入"对外运营"前必须

### 第 2 期：数据洞察（4~6 周）
- **F04 多维退货报表** —— 把"录入工具"升级为"退货洞察平台"

### 第 4 期：平台化（远期，按需）

## 七、重点功能详细可行性

> 以下仅对**第 1 期 P0/P1**功能给出技术可行性细节，使工程团队能据此评估并启动。

### 7.1 F01 用户登录与权限体系

**问题**：当前完全开放访问，任何拿到域名者可使用所有功能；退货记录含客户姓名与电话，存在严重数据安全风险。

**方案要点**：
- 引入轻量级鉴权：Spring Security 6 + JWT（无状态，契合前后端分离 + Docker 部署）
- 角色模型：客服（录入）/ 主管（审核+统计）/ 管理员（用户管理）
- 数据库新增 `sys_user`、`sys_role`、`sys_user_role` 三张表（不改动 return_records / ocr_log 结构）
- 前端增加登录页 + 路由守卫 + 401 拦截；axios 统一注入 `Authorization: Bearer`
- 飞书 OAuth 单点登录可作为加分项（公司已有飞书，免再记一套密码）

**对现有系统的影响**：
- 新增 `security` 包、`SecurityConfig`、`JwtFilter`
- 现有 Controller 加 `@PreAuthorize` 注解；`/api/upload/**` 等接口需登录
- 前端 `api.js` 增加请求/响应拦截器
- Docker 部署无需变更；新增一个 JWT 密钥环境变量 `JWT_SECRET`

**风险**：中
- SSE 接口的鉴权需特别处理（fetch 不能用 axios 拦截器，需要把 token 拼到 URL 或放到 Header）
- 飞书多维表格的写入权限与系统用户权限边界需明确

**验收要点**：未登录访问任意接口返回 401；客服无法访问用户管理页；管理员可增删用户；JWT 过期自动刷新。

---

### 7.2 F02 个人隐私数据脱敏与保留期管理

**问题**：`rec_phone` / `sender_phone` 明文存 MySQL 且参与 DeepSeek 分析，存在个人信息合规风险（PIPL《个人信息保护法》）。

**方案要点**：
- 入库前对电话做**不可逆掩码**：`138****8888`（展示态）+ 单独加密列存完整号（仅在调快递查询等必要时解密）
- LLM 分析时只传**脱敏后的姓名 + 地址前缀**，不传完整电话
- 新增 `data_retention_days` 配置，定时任务清理超期记录（默认 180 天）
- 飞书写入也同步脱敏，避免数据外泄第二出口

**对现有系统的影响**：
- `return_records` 字段含义不变，但写入逻辑改为脱敏入库
- 新增 `DataMaskUtil`、`RetentionCleanupJob`
- 需要决定是否对历史数据回填脱敏（建议一次性脚本处理）

**风险**：中
- 一旦脱敏存储，需要"完整电话"的下游场景（如对接快递 API）需走解密通道
- 飞书多维表格已有历史明文数据，需要评估是否回写

**验收要点**：DB 与飞书均不见明文 11 位手机号；超期记录自动归档/删除并有审计。

---

### 7.3 F11 运单号去重 + 快递公司格式校验

**问题**：同一运单重复上传会重复写飞书，造成脏数据；express_company 与 waybill_no 不匹配（如顺丰号填了"中通"）会误导后续对接。

**方案要点**：
- 上传接口在 OCR 后追加一步：以 `waybill_no` 为准查 `return_records`，若已有 synced 记录则前端弹提示"该运单已入库"，避免重复写飞书
- 新增 `WaybillValidator`：根据快递公司前缀规则校验（如 SF 顺丰、JD 京东、YT 圆通等）
- 校验不通过时返回 `warning`，提示客服人工核对但不阻断流程（保持低摩擦）

**对现有系统的影响**：
- `UploadController.uploadSSE` 流程在"交叉验证"后、"返回结果"前增加"去重 + 校验"步骤
- `return_records` 加唯一索引 `uk_waybill_no`（仅在 status ∈ {confirmed, synced} 时生效，需用部分索引或应用层判定）

**风险**：低
- 仓库偶尔会"重用运单号"（理论上不应发生但要兜底），需保留人工覆盖入口

**验收要点**：同一运单重复上传被拦截并提示；运单号与快递公司明显不匹配时给 warning；不阻塞人工确认。

---

### 7.4 F12 异常监控与飞书告警

**问题**：双引擎 OCR 全失败、飞书写入连续失败、COS 上传失败目前都是"沉默错误"，运维只能事后看日志。

**方案要点**：
- 复用已有飞书机器人 Webhook（项目已集成），新增 `AlertService.notify(level, msg)`
- 触发条件：
  - OCR 双引擎均失败率 > 阈值（如 10% 滑动窗口）
  - 飞书写入连续失败 ≥ 3 次
  - COS 上传失败
  - 应用启动/异常崩溃
- 告警去重：同类告警 5 分钟内只发一次，避免轰炸

**对现有系统的影响**：
- 新增 `AlertService`，在 `GlobalExceptionHandler` 与各 Service 的失败分支中调用
- 不改任何表结构
- Docker 部署无需变更

**风险**：低
- 唯一风险是告警风暴；通过去重 + 级别分级控制

**验收要点**：模拟 OCR 全失败能收到飞书机器人告警；5 分钟内同类告警不重复。

## 八、不建议立即做的功能（及原因）

| 功能 | 原因 |
|------|------|
| 大批量异步队列（F14） | 当前批量接口 300s 超时可覆盖日百单；待日均超千单再投入更划算 |
| 后台配置中心（F13） | 调提示词/阈值属低频操作，环境变量 + 重启成本可接受；过早做配置中心投入产出比低 |
| 对接快递公司真实运单 API | 涉及各家快递商务授权，工程与商务成本双高；建议先看是否真有业务诉求 |
| 飞书小程序入口 | Web 端响应式已能覆盖移动场景；只有"客服必须在飞书内闭环"才值得做 |
| 数据归档自动化 | 数据量未到瓶颈，先观察增长曲线再决定 |

> 原则：**不为可能出现的需求提前建复杂系统**，先让现有主干跑稳、跑出价值。

## 九、与现有项目结构的关系（关键）

为打消"加入这些功能会不会破坏现有架构"的顾虑，明确如下边界：

| 演进项 | 改动边界 | 是否影响主干流程 |
|-------|---------|----------------|
| F01 登录权限 | 新增 `security` 包 + Controller 加注解 | 否（仅加前置校验） |
| F02 脱敏 | 新增 `DataMaskUtil` + 写入处替换 | 是（写入逻辑替换，但字段不变） |
| F11 去重 | upload SSE 流程插一步 | 是（在结果返回前） |
| F12 告警 | 新增 `AlertService` + 失败分支调用 | 否（只增不改） |
| F04~F07 报表 | 新增 `report` 包 + 前端新页 | 否（纯查询，不动写入） |
| F08 字典 | 新增 `sys_dict` 表 + LLM Prompt 注入 | 是（LLM 提示词改造） |
| F09 标注回流 | 新增 `llm_feedback` 表 + 离线评估 | 否（只读 return_records） |

> 结论：**P0/P1 全部可在不破坏现有主干的前提下增量落地**，符合 AGENTS.md 中"文档服务于业务"的反思规则。

## 十、落地建议（给到决策者）

1. **先开会而不是先写代码**：把本方案作为需求评审材料，与业务方、客服主管、IT 同步一次，确认真实优先级。
2. **第 1 期（安全底座）建议与本次 Docker 部署一起上**：既然已决定对外上线，安全与监控"当天补齐"成本最低。
3. **第 2 期（报表）做完再决定要不要做第 3 期**：报表会暴露"哪些退货原因分类不准"，从而决定 F08/F09 是否真的必要，避免过度设计。
4. **每期都走 AGENTS.md 既定的"检查点 + 自审 + 文档同步"流程**：新增表/字段需走"先讨论后改 DDL"约定，本文档列出的新增表均需评审后再动 schema.sql。
5. **本文档与 02~09 同级**，建议在 AGENTS.md「联系文档」表新增一行：`产品演进 -> docs/10-产品演进与功能可行性方案.md`（如你确认，我可以补上）。
