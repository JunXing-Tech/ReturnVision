## 〇、产品定位与不可豁免底线（v1.1 新增）

> 本节为 v1.1 新增的核心约束，明确产品定位与企业级底线，优先级高于本文档其他章节的"原假设"。

### 0.1 产品定位

- **产品形态**：**企业级 SaaS 工具**（非个人自用），面向企业退货录入场景
- **部署形态**：对外 HTTPS 服务，企业多人使用
- **核心目标**：从"录入工具"演进为"退货运营平台"

### 0.2 不可豁免的企业级底线

产品对外上线后，以下 5 项是企业使用的最低门槛，**必须完整落地**：

1. **鉴权**（F01）：企业数据不能裸奔
2. **隐私脱敏**（F02）：客户姓名/电话涉及 PIPL
3. **运单去重**（F11）：多人录入重复是数据灾难
4. **异常告警**（F12）：线上故障必须可感知
5. **操作审计**（F03）：企业出问题要追责

### 0.3 落地原则

- **必要的工作都要做**：不因"工期紧"而妥协方案完整性，按企业级标准交付
- **分阶段而非阉割**：拆期是为了控制风险和验证节奏，不是砍功能；每期的功能都按完整方案落地
- **文档服务于业务**：方案可随业务变化迭代，但已确认的底线不可降级

## 七、重点功能详细可行性

> 以下仅对**第 1a/1b 期 P0**功能给出技术可行性细节，使工程团队能据此评估并启动。
>
> v1.1 更新：修正 F01 的 SSE 鉴权说法（fetch 可设 Header，非不能用）；F11 补充"应用层查重"落点说明，与 v1.1 已加的 OCR 空数据校验对齐。

### 7.1 F01 用户登录与权限体系

**问题**：当前完全开放访问，任何拿到域名者可使用所有功能；退货记录含客户姓名与电话，存在严重数据安全风险。

**方案要点**：
- 引入轻量级鉴权：Spring Security 6 + JWT（无状态，契合前后端分离 + Docker 部署）
- 角色模型：客服（录入）/ 主管（审核+统计）/ 管理员（用户管理）
- 数据库新增 `sys_user`、`sys_role`、`sys_user_role` 三张表（不改动 return_records / ocr_log 结构）
- 前端增加登录页 + 路由守卫 + 401 拦截；axios 统一注入 `Authorization: Bearer`
- 飞书 OAuth 单点登录可作为加分项（公司已有飞书，免再记一套密码）

**对现有系统的影响**：
- 新增 `security` 包、`SecurityConfig`、`JwtFilter`
- 现有 Controller 加 `@PreAuthorize` 注解；`/api/upload/**` 等接口需登录
- 前端 `api.js` 增加请求/响应拦截器
- Docker 部署无需变更；新增一个 JWT 密钥环境变量 `JWT_SECRET`

**风险**：中
- SSE 接口的鉴权：fetch 可正常设置 Header（`Authorization: Bearer xxx`），**只是不能用 axios 拦截器自动注入**，需在 `uploadSSE` 的 fetch 调用里手动加 header，技术上可行
- 飞书多维表格的写入权限与系统用户权限边界需明确

**验收要点**：未登录访问任意接口返回 401；客服无法访问用户管理页；管理员可增删用户；JWT 过期自动刷新。

---

### 7.2 F02 个人隐私数据脱敏与保留期管理

**问题**：`rec_phone` / `sender_phone` 明文存 MySQL 且参与 DeepSeek 分析，存在个人信息合规风险（PIPL《个人信息保护法》）。

**方案要点**：
- 入库前对电话做**不可逆掩码**：`138****8888`（展示态）+ 单独加密列存完整号（仅在调快递查询等必要时解密）
- LLM 分析时只传**脱敏后的姓名 + 地址前缀**，不传完整电话
- 新增 `data_retention_days` 配置，定时任务清理超期记录（默认 180 天）
- 飞书写入也同步脱敏，避免数据外泄第二出口

**对现有系统的影响**：
- `return_records` 字段含义不变，但写入逻辑改为脱敏入库
- 新增 `DataMaskUtil`、`RetentionCleanupJob`
- 需要决定是否对历史数据回填脱敏（建议一次性脚本处理）

**风险**：中
- 一旦脱敏存储，需要"完整电话"的下游场景（如对接快递 API）需走解密通道
- 飞书多维表格已有历史明文数据，需要评估是否回写

**验收要点**：DB 与飞书均不见明文 11 位手机号；超期记录自动归档/删除并有审计。

---

### 7.3 F11 运单号去重 + 快递公司格式校验

**问题**：同一运单重复上传会重复写飞书，造成脏数据；express_company 与 waybill_no 不匹配（如顺丰号填了"中通"）会误导后续对接。

**方案要点**：
- 上传接口在 OCR 后追加一步：以 `waybill_no` 为准查 `return_records`，若已有 synced 记录则前端弹提示"该运单已入库"，避免重复写飞书
- 新增 `WaybillValidator`：根据快递公司前缀规则校验（如 SF 顺丰、JD 京东、YT 圆通等）
- 校验不通过时返回 `warning`，提示客服人工核对但不阻断流程（保持低摩擦）

**对现有系统的影响**：
- `UploadController.uploadSSE` 流程在"交叉验证"后、"返回结果"前增加"去重 + 校验"步骤
- `return_records` 加唯一索引 `uk_waybill_no`（仅在 status ∈ {confirmed, synced} 时生效，需用部分索引或应用层判定）

**v1.1 落点补充**：查重步骤插入位置为 `UploadController` 在 OCR 空数据校验（v1.1 已加的 `BizException(2001)` 处）之后、`recordMapper.insert` 之前，与现有"识别失败不写库"逻辑衔接。

**风险**：低
- 仓库偶尔会"重用运单号"（理论上不应发生但要兜底），需保留人工覆盖入口

**验收要点**：同一运单重复上传被拦截并提示；运单号与快递公司明显不匹配时给 warning；不阻塞人工确认。

---

### 7.4 F12 异常监控与飞书告警

**问题**：双引擎 OCR 全失败、飞书写入连续失败、COS 上传失败目前都是"沉默错误"，运维只能事后看日志。

**方案要点**：
- 复用已有飞书机器人 Webhook（项目已集成），新增 `AlertService.notify(level, msg)`
- 触发条件：
  - OCR 双引擎均失败率 > 阈值（如 10% 滑动窗口）
  - 飞书写入连续失败 ≥ 3 次
  - COS 上传失败
  - 应用启动/异常崩溃
- 告警去重：同类告警 5 分钟内只发一次，避免轰炸

**对现有系统的影响**：
- 新增 `AlertService`，在 `GlobalExceptionHandler` 与各 Service 的失败分支中调用
- 不改任何表结构
- Docker 部署无需变更

**风险**：低
- 唯一风险是告警风暴；通过去重 + 级别分级控制

**验收要点**：模拟 OCR 全失败能收到飞书机器人告警；5 分钟内同类告警不重复。

**v1.1 落点补充**：4 个关键失败分支的埋点位置 -- OCR 双引擎均未识别出有效数据（v1.1 已加的 `BizException(2001)` 处）、飞书写入失败（`confirmSingle` 的 catch 处）、COS 上传失败（`cosClientService.uploadToCos` 抛异常处）、应用启动失败（Spring `ApplicationFailedEvent` 监听）。

## 八、不建议立即做的功能（及原因）

| 功能 | 原因 |
|------|------|
| 大批量异步队列（F14） | 当前批量接口 300s 超时可覆盖日百单；待日均超千单再投入更划算 |
| 后台配置中心（F13） | 调提示词/阈值属低频操作，环境变量 + 重启成本可接受；过早做配置中心投入产出比低 |
| 对接快递公司真实运单 API | 涉及各家快递商务授权，工程与商务成本双高；建议先看是否真有业务诉求 |
| 飞书小程序入口 | Web 端响应式已能覆盖移动场景；只有"客服必须在飞书内闭环"才值得做 |
| 数据归档自动化 | 数据量未到瓶颈，先观察增长曲线再决定 |
| 退货写入成功的群通知（docs/04 4.3.3 FeishuBotService） | 业务价值低（多维表格是真正出口）、群消息轰炸（日百单=百条卡片）、F04 报表是更优解；`feishu.bot-webhook` 当前仅用于 F12 异常告警 |

> 原则：**不为可能出现的需求提前建复杂系统**，先让现有主干跑稳、跑出价值。

## 八.6、数据一致性方案（v1.1 新增）

> 3.2 节提到"飞书写入是最终数据出口，对一致性要求高"，本节展开具体方案。

### 8.6.1 一致性风险点

| 风险 | 触发场景 | 影响 |
|------|---------|------|
| MySQL 与飞书数据不一致 | 飞书写入成功但本地状态更新失败（或反之） | 退货记录页显示"待确认"，飞书已有记录 |
| 飞书写入失败无补偿 | `confirmSingle` 抛异常，记录停留在 confirmed | 数据沉淀在 MySQL，未到飞书，业务方看不到 |
| 飞书侧人工改数据 | 用户在飞书表格直接修改字段 | MySQL 仍是旧值，再次确认会覆盖飞书改动 |
| 重复写飞书 | 同一运单多次确认（F11 未拦截时） | 飞书出现重复行 |

### 8.6.2 最小一致性方案（1a/1b 期落地）

- **MySQL 为主，飞书为辅**：MySQL 是 source of truth，飞书是展示出口；不一致时以 MySQL 为准
- **状态机强约束**：`pending -> confirmed -> synced` 单向流转，`synced` 记录不允许再确认（已在 `confirmSingle` 隐含）
- **飞书写入失败的兜底**：`confirmSingle` 失败时记录留在 `confirmed` 状态，**F12 告警通知**，人工retry
- **幂等保护**：F11 去重 + `feishu_record_id` 回填（已实现），重复确认时检查 `synced` 状态直接返回

### 8.6.3 完整一致性方案（第 3 期及以后）

- **对账任务**：每日凌晨比对 MySQL `synced` 记录与飞书表格行数，不一致告警
- **飞书 Webhook 回流**：监听飞书表格修改事件，回写 MySQL（双向同步，复杂度高）
- **补偿队列**：飞书写入失败入队列，定时重试 3 次，仍失败转人工
- **操作审计**（F03）：所有状态变更记 `operation_log`，便于追溯不一致根因

> v1.1 结论：1a/1b 期只做"MySQL 为主 + 状态机 + F12 告警"最小方案；完整对账和双向同步留到第 3 期，避免过度设计。

## 十、落地建议（给到决策者）

> v1.1 重写：原"先开会"建议基于团队协作假设。在明确企业级 SaaS 定位后，落地建议调整为"按 1a -> 1b -> 2 -> 3 节奏自主推进"。

### 10.1 第一步明确：先做 F12 异常监控告警

**为什么是 F12 而不是 F01 登录？**
1. **风险优先级**：F01 是"防止未来的风险"，F12 是"感知当前的问题"。线上已在跑，**先有眼睛比先有门更重要**
2. **工作量**：F12 是 M 工作量、低风险；F01 是 L 工作量、中风险（SSE 鉴权、历史接口改造）
3. **不阻塞业务**：F12 纯增量，不改任何现有流程；F01 要改所有 Controller + 前端路由，风险大
4. **复用已有基建**：飞书机器人 Webhook 已配好，`AlertService` 几乎是"白捡"的

### 10.2 落地节奏

1. **第 1a 期（1~2 周）**：F12 异常告警 + F11 运单去重 -- 快速止血，让线上"有眼睛 + 不脏数据"
2. **第 1b 期（3~5 周）**：F01 登录权限 + F02 隐私脱敏 + F03 操作审计 -- 安全合规底线
3. **第 2 期（4~6 周）**：F04 报表 + F05 OCR 仪表盘 + F06 导出 -- 数据洞察
4. **第 3 期（6~10 周）**：F08 字典 + F09 标注回流 + F10 图片预检 -- 质量闭环
5. **第 2 期做完再决定要不要做第 3 期**：报表会暴露"哪些退货原因分类不准"，从而决定 F08/F09 是否真的必要，避免过度设计

### 10.3 每期都要遵守的纪律

- **走 AGENTS.md 既定的"检查点 + 自审 + 文档同步"流程**：每个步骤完成后暂停等确认
- **新增表/字段需走"先讨论后改 DDL"约定**：本文档列出的新增表均需评审后再动 schema.sql
- **文档同步**：改表先改 docs/05、改接口先改 docs/06；本文档（docs/10）随演进持续更新版本号
- **代码自审**：每步代码生成后必须自审（编译、逻辑、异常处理、规范、兼容性、业务流程），通过后再进检查点

### 10.4 文档位次

本文档与 02~09 同级，AGENTS.md「联系文档」表已收录：`产品演进 -> docs/10-产品演进与功能可行性方案.md`。
