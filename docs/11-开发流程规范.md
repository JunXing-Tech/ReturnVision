# 11 - 开发流程规范

---

## 〇、文档定位

AGENTS.md 的「执行规则」覆盖的是**编码执行期**（检查点、Git 提交、自审、设计反思、编码约定）。本规范补齐的是它前面的 **"接到一个需求 → 决定动手 → 开发→ 上线"** 这一整段端到端的工作流程，回答两个问题：

1. **先考虑什么，再考虑什么**（流程顺序）。
2. **变更影响面如何评估**（开工前必过的清单）。

> 本规范不替代 AGENTS.md，是其细化的执行手册。两者冲突时以 AGENTS.md 为准。

---

## 一、适用工作类型

| 类型 | 说明 | 是否走本规范 |
|------|------|-------------|
| 新增功能 / 新增模块 | 例如加登录、加报表、加退货字典管理 | ✅ 全流程 |
| 优化功能 | 现有功能改造，如 SSE 加超时、批量加进度 | ✅ 全流程 |
| Bug 修复 | 修复识别异常、写飞书失败等 | ⚠️ 走"影响面评估"+"修复步骤"，可不改文档 |
| 规范/文档调整 | 改本文件、改 AGENTS.md、改基础框架规范 | ⚠️ 走"设计反思规则"，先讨论后改 |
| 纯配置/部署调整 | 改环境变量、改 docker-compose、改 Nginx | ❌ 不走本规范（部署由用户自行处理） |

## 二、变更影响面评估清单（开工前必过）

> 任何功能动手前，**先在工单/PR 描述里回答下列 8 项**，凡出现"是"的项，必须先处理或先讨论：
> **不能跳过这一节直接写代码。**

### 2.1 数据库影响
- [ ] 是否**新增表**？→ 走 AGENTS.md「❌ 修改 DDL SQL」约定：**先讨论**，再改 [docs/05-数据库设计.md](./05-数据库设计.md) 与 `schema.sql`
- [ ] 是否**新增字段 / 索引 / 修改字段**？→ 同上，先讨论
- [ ] 是否需要**回填历史数据**？（如脱敏存量记录）→ 写一次性脚本，并评估执行窗口

### 2.2 API 影响
- [ ] 是否**新增接口**？→ 先补 [docs/06-API接口设计.md](./06-API接口设计.md)，写清请求/响应示例
- [ ] 是否**修改现有接口的请求或响应结构**？→ 评估前后端兼容（前端是否同步改）
- [ ] 是否涉及 **SSE 接口**？→ 前端 fetch 流不能加 axios 拦截器；Nginx 必须 `proxy_buffering off`
- [ ] 是否影响鉴权？→ 与未来 F01 登录体系保持扩展空间

### 2.3 后端影响
- [ ] 在哪些层动手：`controller / service / feishu / model / config / common`？
- [ ] 是否引入**新的第三方依赖**（pom.xml）？→ 评估必要性、版本、License
- [ ] 是否新增**环境变量 / 配置项**？→ 见 2.6
- [ ] 是否触碰**全局异常 / 响应格式**？→ 不能破坏 `ResponseResult` 与 `GlobalExceptionHandler`

### 2.4 前端影响
- [ ] 是否新增页面 / Tab？→ 走 [docs/12-前端设计规范.md](./12-前端设计规范.md) 的 `*Panel.vue` 标准脚手架，并在 `App.vue` 注册
- [ ] 是否修改公共设计 token（`App.vue :root`）？→ 改 token 等于改全站，**先讨论**
- [ ] 是否新增 API 调用？→ 必须在 `src/api.js` 收口，**禁止在组件里裸调 axios**
- [ ] 是否动 SSE？→ 必须用 fetch 流式实现，参考现有 `uploadSSE`
- [ ] 是否新增组件库 / 路由库？→ 默认禁止，确需则先讨论

### 2.5 文档影响（文档先行原则）
- [ ] 哪些 docs 需要同步？对照第六章「文档同步矩阵」逐项确认
- [ ] 是否需要新建 docs？→ 命名沿用 `NN-主题.md`，并在 AGENTS.md「联系文档」表登记

### 2.6 配置影响
- [ ] 是否新增环境变量？→ 三处都要加：
  1. `returnvision-backend/src/main/resources/application.yml`（占位 `${VAR}`）
  2. `deploy/.env.example`（带注释模板）
  3. `deploy/docker-compose.yml`（注入到 backend 的 environment）
- [ ] 业务参数（非密钥）→ 写 `application.yml`；密钥 → 走环境变量，**禁止硬编码**

### 2.7 部署影响
- [ ] 是否需要**重启后端容器**？（改 jar 后 `docker compose up -d --build backend`）
- [ ] 是否需要**重新构建前端 dist**？(`npm run build` 后替换 dist 再重启 frontend 容器)
- [ ] 是否动 Nginx 配置？→ 改 `deploy/nginx/user_conf.d/default.conf`，`docker compose restart frontend`
- [ ] 是否影响证书 / 域名 / 端口？→ 需协调

### 2.8 安全 / 合规影响
- [ ] 是否触及**个人信息**（手机号、姓名、地址）？→ 是否需要脱敏、降权、加保留期
- [ ] 是否引入新对外暴露面？→ 评估是否需要登录/鉴权（参考 docs/10 的 F01）
- [ ] 是否写日志含敏感信息？→ 不能 `log.info` 打手机号明文

> **完成清单后，得到一份"微方案"**：本任务要改哪些层、改哪些文档、是否需讨论 DDL/API、是否需新增环境变量、部署需做什么。把这份微方案作为检查点的提案交出。

## 三、端到端开发步骤（评估通过后执行）

> 前置：第二章清单已过，微方案已成形。下方步骤与 AGENTS.md「检查点」「Git 提交」「自审」「设计反思」**衔接**，不重复其内容。

### 步骤0：需求澄清与边界确认
- 与需求方确认 **What / Why / Who / 验收标准**，未明确前不写代码
- 边界三问：**做不做**？**做到哪**？**不在本次做什么**？

### 步骤1：设计与影响面评估
- 输出微方案，覆盖第二章 8 项清单
- 若发现更好的方案 → 触发 AGENTS.md「设计反思规则」，**先暂停建议**

### 步骤2：文档先行
- 先改 docs（02~09 / 基础框架规范 / 11 / 12），**再写代码**（AGENTS 设计反思规则要求）
- 新增接口先写 [06-API接口设计.md](./06-API接口设计.md)；改表先改 [05-数据库设计.md](./05-数据库设计.md)

### 步骤3：DDL / 数据库变更讨论
- 涉及改表/改字段/改索引 → **先讨论**，通过后再动 `schema.sql` 与实体（AGENTS 禁止事项红线）

### 步骤4：API 契约确定
- 先定 request / response 结构，前后端共用此契约，再各自实现
- 响应一律 `ResponseResult<T>`

### 步骤5：后端实现
- 顺序：`model(entity/mapper) → service → controller`
- 遵循 [基础框架规范.md](./基础框架规范.md) 全部约定（响应、异常、日志、配置、注释）
- Controller 写业务流程注释 + `// 步骤N`；Service 写实现步骤注释
- 异常抛 `BizException` 子类，**不手动 try-catch**

### 步骤6：前端实现
- 遵循 [docs/12-前端设计规范.md](./12-前端设计规范.md)
- 新页面用 `*Panel.vue` 标准脚手架；API 在 `api.js` 收口；样式用 `var(--token)`

### 步骤7：联调自测
- 正常路径 + 边界（空、超长、非法）+ 异常路径（OCR 全失败、飞书限频）+ SSE 流式验证
- 自测覆盖与已有功能交互的回归点

### 步骤8：代码自审（沿用 AGENTS 6 项）
1. 编译是否通过
2. 逻辑正确、边界是否考虑
3. 异常是否走 `BizException`
4. 规范是否符合（注释、日志、构造器注入）
5. 与已有代码的交互兼容（方法签名、返回类型、依赖）
6. 业务流程是否完整顺畅

### 步骤9：检查点暂停 + Git 提交
- 按 AGENTS「检查点规则」暂停，等用户"继续"
- 按 AGENTS「Git 提交规则」提交并推送，提交信息格式：`模块/功能：简要描述`

## 四、按工作类型的快速路径

| 类型 | 走哪些步骤 |
|------|-----------|
| 新增功能/模块 | 步骤0 → 9 全走 |
| 优化功能 | 步骤1 → 9（步骤0 简化为确认目标） |
| Bug 修复 | 第二章清单（聚焦 2.2~2.3）→ 步骤5/6/7/8/9；可不改 docs，但需在 PR 说明根因 |
| 规范/文档调整 | AGENTS「设计反思规则」：先讨论后改 |

---

## 五、强制门禁（必须停下来讨论，不得自行决定）

| 触发条件 | 必做动作 |
|---------|---------|
| 需要改 `schema.sql` / 改表 / 改字段 / 改索引 | 先讨论，通过后再改，并同步 docs/05 |
| 需要改公共设计 token（`App.vue :root`） | 先讨论，评估全站影响 |
| 需要引入新组件库 / vue-router / 状态管理库 | 先讨论，默认拒绝 |
| 需要破坏 `ResponseResult` 或 `GlobalExceptionHandler` 契约 | 先讨论 |
| 需要硬编码密钥 / 把密钥写进 git | 禁止，必须走环境变量 |
| 需要 skip 检查点 | 禁止，按 AGENTS 执行 |

## 六、文档同步矩阵

> "改了 X，必须同步 Y"。这是"开发流程一直没有得到规范"的根因之一：代码变了，文档没跟上。

| 改动类型 | 必须同步的文档 |
|---------|---------------|
| 新增/改表或字段 | docs/05-数据库设计.md、`schema.sql`、`model/entity/*` |
| 新增/改接口 | docs/06-API接口设计.md、前端 `api.js` |
| 新增/改模块 | docs/04-详细方案设计.md、docs/07-项目结构与代码规范.md（如新增包/页面） |
| 改基础约定 | docs/基础框架规范.md、AGENTS.md |
| 新增环境变量 | `application.yml`、`deploy/.env.example`、`deploy/docker-compose.yml` |
| 改前端设计 token / 新页面 | docs/12-前端设计规范.md |
| 改部署 / Nginx / 证书 | deploy/ 目录文件、AGENTS.md 技术栈节如有变 |
| 业务流程变化 | AGENTS.md「业务流程速览」图、docs/01 项目概述 |

> **澄清 → 评估 → 文档先行 → DDL/API → 后端 → 前端 → 联调 → 自审 → 检查点 → 提交**

每一步完成都对应 AGENTS.md 既有规则，不额外加重负担；**省哪一步都可能在后期返工**。
