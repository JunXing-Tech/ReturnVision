# 技术架构设计

### 2.1 技术栈选型

| 层级 | 技术 | 选型理由 |
|------|------|---------|
| **后端语言** | Java 21+ | 企业级稳定性，团队技术栈统一 |
| **后端框架** | Spring Boot 3.x | 成熟生态，适合企业级应用 |
| **OCR 引擎** | 智谱OCR解析工具（引擎A）+ 阿里云电子面单识别（引擎B） | 双引擎交叉验证，智谱94.62%准确率、0.01元/次；阿里云93%、有免费额度及逐字段置信度 |
| **数据库ORM** | MyBatis-Plus | 国内主流，灵活SQL控制，代码生成便捷 |
| **数据库** | MySQL 8.0 | 生产级可靠性，存储待确认记录、OCR日志、状态流转 |
| **图片存储** | 腾讯云 COS | 持久化存储快递单照片，URL供双引擎OCR和飞书引用 |
| **LLM分析** | DeepSeek V4 Flash | 语义校验+退货原因分析+智能分类，~0.0004元/次 |
| **飞书 SDK** | lark-oapi（Java官方SDK） | 写入飞书多维表格 |
| **前端** | Vue 3 + Vite | 轻量，适合简单的上传+确认页面 |

### 2.3 数据流转详解

**完整流程**：
```
1. 客服拍摄快递单照片（手机拍照/截图）
2. 上传到后端（支持单张/批量）
3. 后端将图片上传至腾讯云COS，获取图片URL
4. 后端并行调用双引擎OCR识别（传入URL）：
   ├── 智谱OCR（layout_parsing）-> 结果A
   └── 阿里云面单OCR（RecognizeWaybill）-> 结果B（含valueProb置信度）
5. 交叉验证引擎比对结果A与结果B：
   ├── 关键字段一致 -> 自动采用（覆盖~88%场景，准确率接近99.9%）
   ├── 部分字段不一致 -> 高亮差异字段，用阿里云置信度辅助判断
   └── 完全不一致 -> 转人工录入
6. 运单号格式校验（8-30位字母数字）
7. DeepSeek V4 Flash LLM分析：语义校验 + 退货原因提取 + 智能分类
8. 识别结果 + COS图片URL + LLM分析结果存入MySQL（status=pending）
9. 前端展示识别结果：
   ├── 一致结果：客服快速确认（~2秒）
   ├── 有差异：客服只核对高亮字段（~5秒）
   └── 全失败：客服手动录入（~15秒）
10. 确认后写入飞书多维表格（含photo_url、退货原因、分类），更新MySQL状态（status=synced）
```

### 2.4 五层保障体系

| 层级 | 机制 | 说明 |
|------|------|------|
| 第一层 | 双引擎交叉验证（自动） | 两引擎一致时自动采用，覆盖~88%场景 |
| 第二层 | 阿里云置信度辅助（自动） | valueProb ≥ 80 高置信采用；<80 标记复核 |
| 第三层 | 运单号格式校验（自动） | 8-30位字母数字格式校验，异常标记 |
| 第四层 | DeepSeek语义校验（自动） | LLM检查运单号前缀与快递公司匹配等跨字段逻辑 |
| 第五层 | 人工确认（必经） | 所有结果经客服确认后写入飞书 |
