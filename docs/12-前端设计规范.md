## 〇、文档定位

- v2.x 以 Vercel 单色系为基线；v3.0 起**切换为源力（Volcengine）设计系统**，原因是源力在业务匹配、状态表达、中文可读性三个维度全面优于 Vercel（详见设计评估记录）。
- 本文件**把源力风格已落地的约定固化下来**，作为后续新增页面、优化交互的唯一依据。
- 与 AGENTS.md「编码约定」不冲突处沿用；与本文件冲突处，前端以**本文件**为准。
- **设计哲学**：舒服协调是底线，记忆点是加分项，token 纪律是质量保障。三者冲突时，优先保证协调性和 token 纪律。

## 三、设计系统 Token（来源：源力 Volcengine，prefix `yuanli`）

> 改这一节等于改全站，必须走 docs/11 第五章门禁"先讨论"。
> v3.0 起采用 **源力品牌蓝 + 中文优先 + 完整状态色 ramp** 双主题：默认浅色（运营场景优先），暗色通过 `:root.dark` 覆盖。
> 组件一律用 `var(--yuanli-*)`，**禁止写死色值**--切换主题时组件零改动。

### 3.1 双主题架构

- `:root` 定义浅色为默认值（全量 token）
- `:root.dark` 仅覆盖需要变化的色值（背景面/主色/边框/状态色/侧边栏/图表色/阴影/Element Plus 变量）
- 尺度变量（`--yuanli-radius-*`/`--yuanli-spacing`/`--yuanli-font-*`/`--yuanli-transition`）两主题共享，不重复定义
- 通过给 `<html>` 加/去 `dark` class 切换（见第十五章）

### 3.2 颜色对照（浅色默认 / 暗色覆盖）

### 3.3 状态色用法铁律（v3.0 核心改动）

退货 OCR 工作台有**四个高频核对状态**，状态色必须用「浅底色块徽章 + 强色文字」强反馈，让运营一眼扫描：

| 状态 | 徽章类 | 底色 | 文字色 | 场景 |
|------|--------|------|--------|------|
| 已同步 | `.status-success` | `--yuanli-success-subtle` | `--yuanli-success-strong` | 飞书写入完成 |
| 待确认 | `.status-warning` | `--yuanli-warning-subtle` | `--yuanli-warning-strong` | OCR 识别完成待人工核对 |
| 识别失败 | `.status-danger` | `--yuanli-danger-subtle` | `--yuanli-danger-strong` | 双引擎识别失败 |
| 处理中 | `.status-info` | `--yuanli-info-subtle` | `--yuanli-primary` | SSE 流式处理中 |

**铁律**：
- 状态徽章用胶囊形（`border-radius: var(--yuanli-radius-full)`），padding 小，字号 11px
- `*-subtle` 浅底 + `*-strong` 文字，**禁止**用纯状态色做整块背景填充（如整行高亮、整块红卡）
- 状态色 ramp 必须成对使用（subtle 底配 strong 字），不可混用主题

### 3.4 排版与尺度（两主题共享）
- 圆角：`--yuanli-radius-sm: 4px`（控件）/ `--yuanli-radius-md: 8px`（默认）/ `--yuanli-radius-lg: 12px`（卡片）/ `--yuanli-radius-xl: 16px`（大卡片，上限）/ `--yuanli-radius-full: 9999px`（胶囊徽章）
- 间距单位：`--yuanli-spacing: 4px`，间距一律用 `calc(var(--yuanli-spacing) * N)` 或 `var(--yuanli-space-N)`
- 阴影：5 级 `--yuanli-shadow-sm/md/lg/xl`（半透明黑 0.03~0.10），暗色覆盖为 0.3~0.5；focus 用 `--yuanli-shadow-blue`
- 过渡：`--yuanli-transition: 150ms ease`
- 字体：`--yuanli-font-sans: "PingFang SC", "Microsoft YaHei", "Roboto", "Helvetica Neue", Arial, sans-serif`（**中文优先**）
- 等宽：`--yuanli-font-mono: "SFMono-Regular", "SF Mono", "Roboto Mono", "Consolas", monospace`
- 基准字号：`--yuanli-font-size-body: 13px`（密集工作台风格）
- 字重：regular 400 / medium 500 / semibold 600（标题用 600）
- 标题字距：大标题加 `letter-spacing: -0.01em`

### 3.5 数据字体强制规则

运单号、ID、电话、时间戳、数字**一律用 `var(--yuanli-font-mono)`**，强化"数据 vs 叙述"层级。这是源力风格区别于普通后台的关键，不可妥协。

### 3.6 Element Plus 主题覆盖
两主题分别在 `:root` 和 `:root.dark` 覆盖 `--el-*` 变量，关键项：
```css
/* 浅色 */
--el-color-primary: #1664ff;
--el-bg-color: #ffffff;
--el-text-color-primary: #0c0d0e;
--el-border-color: #dde2e9;
--el-fill-color: #f6f8fa;

/* 暗色（:root.dark 覆盖） */
--el-color-primary: #fafafa;
--el-bg-color: #0c0d0e;
--el-text-color-primary: #ffffff;
--el-border-color: #2a3440;
--el-fill-color: #1d2129;
```
> 禁止用 `::v-deep` 大面积改 Element Plus 组件样式；能用变量覆盖的优先用变量。
> 两个主题块都要覆盖 `--el-fill-color-*`、`--el-mask-color`、`--el-box-shadow` 等浮层变量。

## 四、页面与组件粒度

### 4.1 外壳统一 `screen-shell`
每个 `*Panel.vue` 顶层模板：
```html
<div class="screen-shell">
  <section class="hero"> ... </section>
  <section class="card"> ... </section>
</div>
```

### 4.2 内部分区习惯（参考 `DashboardPanel.vue`）
- `hero`：标题 + 主操作按钮（品牌蓝 primary CTA）
- `kpi-grid`：关键指标卡片网格（数字用 24px 600 字重，带 sparkline）
- `main-content`：主内容区（左表 + 右图，或左右双列）
- `card`：分组容器，内含 `card-header` + 内容，白底 + 边框 + shadow-md
- `toolbar`：筛选/搜索/刷新

### 4.3 多列布局平衡铁律（v3.0 新增，踩坑教训）

**多列布局必须检查列高对齐，禁止出现"一列过长一列过短"的失衡。**

踩坑案例：面单识别页曾用「左 3 卡纵向堆叠 / 右 1 卡表单」，左列高度远超右列，视觉失衡。修正方案：改为「左 2 卡 / 右 2 卡」平衡结构。

规则：
- 双列布局前，预估两列内容高度，差值不超过 20%
- 若一列内容明显多于另一列，把部分卡片移到另一列（如把"字段校验"从证据列移到表单列上方）
- `grid-template-columns` 比例用 `1fr 1fr` 或 `1fr 1.05fr`（接近等宽），避免 `1fr 1.5fr` 等悬殊比例
- 用 `align-items: start` 让各列从顶部对齐

### 4.4 路由切换
- `App.vue` 维护 `activeTab`，三种 Tab 用 `v-show` 切换（**不用 v-if**，保留上传进度等状态）
- 跨页传值：`pendingEditRecord` 等 `ref`，通过 props/emit 携带
- 顶栏"搜索框"目前为占位，接入搜索功能时统一抽到 `composables/`

## 六、样式书写约定

| 项 | 约定 |
|----|------|
| 颜色 | 一律 `var(--yuanli-*)`，**禁止硬编码 `#xxxxxx`** |
| 间距 | 一律 `calc(var(--yuanli-spacing) * N)` 或 `var(--yuanli-space-N)`，**禁止魔法数 px** |
| 圆角 | 控件 `var(--yuanli-radius-sm/md)`，卡片 `var(--yuanli-radius-lg/xl)`，徽章 `var(--yuanli-radius-full)` |
| 布局 | 优先 Flex / Grid + 显式 gap，少用 float/position/absolute |
| 作用域 | 组件 `<style>` 默认 scoped；全局样式只放 `App.vue` 或 `tokens.css` |
| 响应式 | 通过 `useMobile` + `@media` 断点（`1080px`、`900px`、`640px`） |
| 动效 | 用 `var(--yuanli-transition)`，禁止花哨动画；加载用 spinner |
| 数据字体 | 运单号/数字/时间戳用 `var(--yuanli-font-mono)`，强化「数据 vs 叙述」层级 |
| 标题字距 | 大标题（hero-title/card-title 等）加 `letter-spacing: -0.01em` |
| 焦点态 | input focus 用 `box-shadow: 0 0 0 3px rgba(56,123,255,0.15)`（品牌蓝 ring） |

> **rgba 透明度例外**：focus ring 的 `rgba(56,123,255,0.15)` 是品牌蓝 `--yuanli-ring`（#387bff）加 15% 透明度的表达，CSS 变量无法单独承载透明度，此处 rgba 属于合理例外。其他场合一律用 `var(--yuanli-*)`，禁止硬编码色值。
> **不得新增全局样式文件**（如 `main.css`、`theme.scss`），所有 token 集中在 `App.vue :root` 或 `tokens.css`。

### 6.1 Token 语义反转禁令（v3.0 新增，踩坑教训）

**禁止把前景色 token 当背景用，或把背景色 token 当文字用。**

踩坑案例：登录页重设计曾用 `--yuanli-foreground`（#0c0d0e 本是文字色）当深色背景，`--yuanli-background`（#ffffff 本是背景色）当文字。颜色值虽然对，但违背语义约定，导致：
1. 后续维护混乱（看 token 名猜不出用途）
2. 无法响应全局暗色模式切换（dark 覆盖会反转语义）
3. 与其他页面主题割裂

规则：
- `--yuanli-background` / `--yuanli-card` / `--yuanli-muted` / `--yuanli-color-bg-*` 只能做背景
- `--yuanli-foreground` / `--yuanli-muted-foreground` / `--yuanli-color-text-*` 只能做文字/图标色
- 若需局部深色区域，在页面 `<style>` 内定义局部变量引用已有深色 token（如 `--page-deep: var(--yuanli-color-bg-2)`），**不可反转语义**

## 七、API 调用约定

### 7.1 收口
- **所有请求集中在 `src/api.js`**，组件只调用其导出方法，**禁止在组件里裸 import axios / fetch**
- `api.js` 内 `axios` 实例 `baseURL='/api'`，超时按场景设（普通 120s、批量 300s）

### 7.2 响应解包
后端统一 `ResponseResult<T>`，但前端历史代码同时读驼峰与下划线（见 RecordsPanel），新代码请：
- 优先读取后端返回的 `data` 字段并直接消费
- 字段命名以后端实体为准，前端展示层做兼容映射，不脏化全局

### 7.3 SSE（流式上传）
- 必须用**原生 `fetch`** 读 `ReadableStream`，按 `data:` 行解析（见 `api.js` 的 `uploadSSE`）
- 提供 `onStep / onResult / onError` 回调与 `AbortController` 取消
- **不要**用 axios 发 SSE；鉴权一旦接入，token 走 Header 或 URL 拼参（fetch 不能用 axios 拦截器）

### 7.4 错误处理
- 统一在 `api.js` 不捕获，由调用方在 `try/catch` 或 `.catch` 处理，给出 toast/inline 错误
- 业务失败展示后端 `msg`；网络错误给友好文案

## 八、表单 / 校验 / 提交态

- 用 `el-form` + `:model` + `:rules`，校验规则集中在组件 `const rules = {...}`
- 提交按钮必须有 loading 态（`:disabled="submitting"` + spinner）
- 关键操作（删除、批量确认）必须二次确认（`ElMessageBox.confirm`）
- 成功/失败用 `ElMessage` 反馈，与 `--yuanli-success / --yuanli-danger` 视觉一致
- input focus 态用品牌蓝 ring（`box-shadow: 0 0 0 3px rgba(56,123,255,0.15)`）
- 主提交按钮用品牌蓝 primary（`--yuanli-primary` 底 + `--yuanli-primary-foreground` 字）

## 九、表格约定（密集工作台风格）

- 容器 `table-wrap` + `data-table`，密集行高，13px 字号
- 状态列用「浅底色块徽章 + 强色文字」（源力风格，见 3.3 状态色铁律）：
  - 已同步：`.status-success`（success-subtle 底 + success-strong 字）
  - 待确认：`.status-warning`（warning-subtle 底 + warning-strong 字）
  - 识别失败：`.status-danger`（danger-subtle 底 + danger-strong 字）
- 运单号列一律 `font-family: var(--yuanli-font-mono)`，强化数据感
- 表头用 `text-transform: uppercase; letter-spacing: 0.05em` 小标签风格，颜色 `var(--yuanli-color-text-3)`
- 行 hover 用 `var(--yuanli-color-bg-5)` 叠加，禁止整行状态色高亮
- 工具栏统一结构：状态筛选（pill-toggle）+ 搜索框（`search-shell`）+ 刷新（`icon-btn`）
- 大量数据后续若超 500 行再考虑虚拟滚动，先不过度设计

## 十、响应式与移动端

- 断点：`≤1080px` 收窄搜索框；`≤900px` 隐藏侧栏，单列布局；`≤640px` 卡片全宽、padding 减小
- 复用 `composables/useMobile.js`
- 移动端上传需触发系统相机：`<input accept="image/*" capture="environment">`，按需开启

## 十一、可访问性（最低要求）

- 按钮用 `<button>` 而非 `<div @click>`，保留键盘可达
- 图标按钮加 `aria-label` 或旁置文案
- 颜色不作为唯一信息载体（状态除色块徽章外必须有文字）
- 表单 label 与控件绑定
- 焦点态必须有可见 ring（品牌蓝 `--yuanli-ring`）

## 十二、新增页面的标准脚手架

> 新增页面时，复制此骨架起步，再按业务填充，避免结构游离。

```vue
<template>
  <!-- 步骤1：页面外壳 -->
  <div class="screen-shell">
    <!-- 步骤2：Hero -->
    <section class="hero">
      <div class="eyebrow">分类</div>
      <h1 class="hero-title">页面标题</h1>
      <p class="hero-subtitle">一句话说明</p>
    </section>

    <!-- 步骤3：主内容卡片 -->
    <section class="card">
      <div class="card-header">
        <div class="card-heading">
          <div class="eyebrow">数据</div>
          <h3 class="card-title">主标题</h3>
        </div>
        <div class="toolbar">
          <!-- 筛选 / 搜索 / 刷新 -->
        </div>
      </div>
      <div v-loading="loading" class="table-wrap">
        <!-- 表格或内容 -->
      </div>
    </section>
  </div>
</template>

<script setup>
// 步骤4：状态与 API 调用
import { ref, onMounted } from 'vue';
import api from '../api';

const loading = ref(false);
const list = ref([]);

// 步骤5：加载方法
const load = async () => {
  loading.value = true;
  try {
    const { data } = await api.getRecords();
    list.value = data?.list || [];
  } catch (e) {
    // 错误文案展示
  } finally {
    loading.value = false;
  }
};

onMounted(load);
</script>

<style scoped>
/* 步骤6：组件作用域样式，引用源力 token */
.foo { padding: calc(var(--yuanli-spacing) * 4); border-radius: var(--yuanli-radius-md); color: var(--yuanli-foreground); background: var(--yuanli-card); border: 1px solid var(--yuanli-border); }
.mono { font-family: var(--yuanli-font-mono); }
.status-success { background: var(--yuanli-success-subtle); color: var(--yuanli-success-strong); border-radius: var(--yuanli-radius-full); padding: 2px 8px; font-size: 11px; }
</style>
```

注册到 `App.vue`：
```js
import NewPanel from './components/NewPanel.vue';
const tabs = [ ..., { key: 'new', label: '新页面', icon: SomeIcon }];
// 模板：<NewPanel v-show="activeTab === 'new'" />
```

## 十三、设计哲学三原则（v3.0 新增）

> 本节是 v3.0 的灵魂，所有设计决策必须通过这三原则检验。

### 13.1 舒服协调是底线
- 全站主题统一（浅色默认），禁止局部深色区域割裂主题
- 多列布局列高平衡（见 4.3）
- 卡片语言一致：白底 + 边框 + shadow-md + 12px 圆角，全站复用
- 排版注重留白和呼吸感，padding 充足（卡片 20-36px，元素间距 16-24px）
- 字号比例舒适（标题 20px / 副标题 13px / 正文 13px / 标签 11px）

### 13.2 记忆点是加分项
- 品牌蓝 `#1664ff` 作为唯一视觉锚点（主 CTA / 激活态 / focus ring / 链接）
- 状态色块徽章强反馈，让运营一眼扫描核对状态
- 数据字体（mono）强化运单号/ID/时间戳的数据感
- 图表 5 色区分退货趋势系列
- 记忆点不可破坏协调性：若某设计有记忆点但与项目割裂，放弃记忆点保协调

### 13.3 Token 纪律是质量保障
- 所有颜色用 `var(--yuanli-*)`，禁硬编码
- 禁语义反转（见 6.1）
- 状态色 ramp 成对使用（subtle 底配 strong 字）
- 新增 token 必须同时在 `:root` 和 `:root.dark` 定义
- 验证脚本（scan-design-directory）是最后质量门，0 错误才能交付

## 十四、禁止事项（前端专属）

| 禁止 | 原因 |
|------|------|
| ❌ 硬编码颜色 `#xxxxxx` | 必须用 `var(--yuanli-*)` token |
| ❌ 语义反转（foreground 当背景用） | 违背 token 语义，破坏主题切换（见 6.1） |
| ❌ 局部深色区域割裂主题 | 全站主题统一，深色需求走全局暗色模式 |
| ❌ 玻璃拟态（backdrop-filter）做主视觉 | 与务实 B 端工作台风格不符 |
| ❌ 多列布局列高失衡 | 必须检查列高对齐（见 4.3） |
| ❌ 引入新组件库 / 路由库 / 状态库 | 默认拒绝，先讨论 |
| ❌ 用 `v-if` 切主路由 Panel | 应 `v-show` 保留状态 |
| ❌ 跳过模板"步骤N"注释 | 与后端注释规范一致 |
| ❌ 用 SSE 加 axios 拦截器 | fetch 流式与 axios 拦截器不兼容，SSE 必须用原生 fetch |
| ❌ 全局样式散落到多个文件 | token 集中 `App.vue :root` 或 `tokens.css` |
| ❌ 状态色用纯色整块背景填充 | 只用 `*-subtle` 浅底 + `*-strong` 文字徽章 |
| ❌ 运单号/数字用正文字体 | 必须用 `var(--yuanli-font-mono)` |
| ❌ 组件里写死色值（不区分主题） | 必须用 `var(--yuanli-*)`，否则暗色下错色 |
| ❌ 用 JS 直接操作 `--yuanli-*` 变量切主题 | 必须通过 `<html>` 加/去 `dark` class |
| ❌ 在组件里裸用 axios / fetch | 必须经 `api.js` 收口 |
| ❌ 魔法数 px 间距 | 必须用 `calc(var(--yuanli-spacing) * N)` 或 `var(--yuanli-space-N)` |

## 十五、主题切换规范

### 15.1 架构
- 主题切换通过给 `<html>` 加/去 `dark` class 实现，触发 `:root.dark` CSS 覆盖
- 禁止用 JS 直接修改 `--yuanli-*` 变量值；禁止在组件内写 `@media (prefers-color-scheme)` 分支
- 组件一律用 `var(--yuanli-*)`，切换时组件零改动

### 15.2 composable
- 主题逻辑集中在 `src/composables/useTheme.js`，组件只调用 `useTheme()` 取 `isDark` 和 `toggleTheme`
- 重复调用 `useTheme()` 安全（内部 `initialized` 守卫，只初始化一次）

### 15.3 首次访问策略
- **localStorage 优先**：键名 `returnvision-theme`，值为 `'light'` 或 `'dark'`
- 未保存时**跟随系统** `prefers-color-scheme`
- 用户手动切换后写入 localStorage，之后不再跟随系统变化

### 15.4 防闪屏（FOUC）
- `main.js` 在 `createApp` 前提前读取主题并应用 `dark` class，避免挂载前的浅色闪屏
- 该逻辑与 `useTheme.js` 的 `resolveInitialTheme` 保持一致，改动时两处同步

### 15.5 切换按钮
- 位置：顶栏右侧（搜索框旁的工具栏），与通知铃铛同排
- 图标：浅色态显示月亮（Moon），暗色态显示太阳（Sun）
- 样式复用 `.icon-btn`，hover 用 `var(--yuanli-accent)`，两主题自动跟随
- 必须有 `aria-label`（"切换到浅色"/"切换到暗色"），保证可访问性

### 15.6 新增主题色值的原则
- 两主题下都要保证文字与背景对比度（WCAG AA：正文 ≥ 4.5:1）
- 暗色下状态色需提亮（如 success `#309256` -> `#7ccd94`），避免暗底上发闷
- 新增 token 必须同时在 `:root` 和 `:root.dark` 定义，缺一会在某主题下回退到默认值

## 十六、CTA 与按钮规范（v3.0 新增）

| 类型 | 样式 | 用途 |
|------|------|------|
| Primary | `--yuanli-primary` 底 + `--yuanli-primary-foreground` 字 | 主操作（上传识别、确认写入飞书、登录） |
| Secondary | `--yuanli-color-bg-4` 底 + `--yuanli-border` 边框 | 次操作（筛选、查看） |
| Ghost | 透明底 + `--yuanli-border` 边框 | 辅助操作（重新上传、返回） |
| Danger | `--yuanli-danger` 底 + 白字 | 危险操作（删除） |

- 高度默认 34px，大号 40px
- 字号 13px，字重 500
- 圆角 `--yuanli-radius-md`（8px）
- hover 态：Primary 变 `--yuanli-ring`，Secondary/Ghost 变 `--yuanli-color-bg-5`
- 一行最多 1 个 Primary，避免主操作焦点分散
