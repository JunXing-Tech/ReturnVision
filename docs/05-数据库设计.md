# 数据库设计

#### 4.5.1 为什么需要本地数据库

退货OCR的最终数据写入飞书多维表格，但本地数据库在**识别到确认**这段流程中不可或缺：

| 理由 | 说明 |
|------|------|
| 待确认记录暂存 | OCR 识别后、客服确认前，需持久化识别结果；客服可能关闭浏览器后回来继续 |
| 状态流转管理 | 记录从"待确认 -> 已确认 -> 已同步飞书"的全生命周期状态 |
| 识别质量监控 | 记录 OCR 引擎、置信度、耗时，用于监控准确率、优化引擎选择 |
| 历史查询加速 | 本地缓存退货记录，查询历史不必每次调飞书 API |

#### 4.5.2 表结构设计

本项目设计 **2 张表**，精简够用：

**表1：return_records（退货记录表）** - 核心表

```sql
CREATE TABLE return_records (
    id              INT PRIMARY KEY AUTO_INCREMENT,
    waybill_no      VARCHAR(30),                    -- 运单号
    rec_name        VARCHAR(50),                    -- 收件人姓名
    rec_phone       VARCHAR(20),                    -- 收件人电话
    rec_address     TEXT,                           -- 收件人地址
    sender_name     VARCHAR(50),                    -- 寄件人姓名
    sender_phone    VARCHAR(20),                    -- 寄件人电话
    express_company VARCHAR(30),                    -- 快递公司
    return_date     DATE,                           -- 退货日期
    status          VARCHAR(20) DEFAULT 'pending',  -- 状态：pending/confirmed/synced/failed
    ocr_engine      VARCHAR(20),                    -- 识别引擎：zhipu/aliyun/cross_validated/manual
    ocr_confidence  DECIMAL(3,2),                   -- OCR置信度（0.00-1.00）
    image_url       VARCHAR(500),                   -- 腾讯云COS图片URL
    return_reason   VARCHAR(100),                   -- 退货原因（DeepSeek分析）
    return_category VARCHAR(50),                    -- 退货分类：质量问题/物流问题/尺寸不符/...
    llm_confidence  DECIMAL(3,2),                   -- LLM分析置信度（0.00-1.00）
```

**表2：ocr_log（OCR识别日志表）** - 识别监控

```sql
CREATE TABLE ocr_log (
    id            INT PRIMARY KEY AUTO_INCREMENT,
```

#### 4.5.3 状态流转设计

return_records 的 status 字段管理记录全生命周期：

```
[上传图片] -> [OCR识别]
                  │
        ┌─────────┴─────────┐
        ▼                   ▼
   识别成功              识别失败
   status=pending       status=failed
        │                   │
        ▼                   ▼
   客服确认/修改        手动录入
   status=confirmed     status=confirmed
        │                   │
        ▼                   ▼
   写入飞书表格
   status=synced
   回填 feishu_record_id
```

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| pending | 待确认 | OCR识别成功，等待客服确认 |
| confirmed | 已确认 | 客服确认或手动录入 |
| synced | 已同步 | 已写入飞书多维表格 |
| failed | 识别失败 | OCR引擎均无法识别 |

#### 4.5.4 ER 关系图

```
┌─────────────────────┐         ┌─────────────────────┐
│  return_records      │         │  ocr_log             │
│─────────────────────│         │─────────────────────│
│  id (PK)            │◀────────│  record_id (FK逻辑)  │
│  waybill_no         │  一对多  │  engine              │
│  rec_name           │         │  duration_ms         │
│  status             │         │  success             │
│  ocr_engine         │         │  confidence          │
│  ocr_confidence     │         │  error_msg           │
│  image_url          │         └─────────────────────┘
│  return_reason      │
│  return_category    │
│  llm_confidence     │
│  feishu_record_id   │
│  created_at         │
│  confirmed_at       │
│  synced_at          │
└─────────────────────┘
```

> 一条退货记录会同时经历两个引擎OCR（智谱+阿里云并行），因此 ocr_log 与 return_records 为一对多关系。

#### 4.5.8 鉴权表设计（F01，v2.1 新增）

> 关联：[docs/10-产品演进与功能可行性方案.md](./10-产品演进与功能可行性方案.md) 第 7.1 节 F01
> 状态：✅ DDL 设计已确认（2026-07-18）
> 设计原则：RBAC 标准模型，4 表分离（用户/角色/关联/refresh token），不改动 return_records / ocr_log 结构

**表3：sys_user（用户表）**

```sql
CREATE TABLE IF NOT EXISTS sys_user (
    id              INT PRIMARY KEY AUTO_INCREMENT,
    username        VARCHAR(50) NOT NULL UNIQUE,         -- 登录用户名
    password_hash   VARCHAR(100),                        -- BCrypt 哈希（飞书 OAuth 用户可为空）
    display_name    VARCHAR(50),                         -- 显示名称
    feishu_user_id  VARCHAR(50),                         -- 飞书 user_id（OAuth 绑定用，可为空）
    status          VARCHAR(20) DEFAULT 'active',        -- active/disabled
    last_login_at   DATETIME,                            -- 最后登录时间
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE INDEX idx_sys_user_feishu ON sys_user(feishu_user_id);
```

**表4：sys_role（角色表）**

```sql
CREATE TABLE IF NOT EXISTS sys_role (
    id           INT PRIMARY KEY AUTO_INCREMENT,
    role_code    VARCHAR(30) NOT NULL UNIQUE,            -- STAFF/SUPERVISOR/ADMIN
    role_name    VARCHAR(30) NOT NULL,                   -- 客服/主管/管理员
    description  VARCHAR(100),
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**表5：sys_user_role（用户-角色关联表）**

```sql
CREATE TABLE IF NOT EXISTS sys_user_role (
    user_id  INT NOT NULL,
    role_id  INT NOT NULL,
    PRIMARY KEY (user_id, role_id),
    FOREIGN KEY (user_id) REFERENCES sys_user(id) ON DELETE CASCADE,
    FOREIGN KEY (role_id) REFERENCES sys_role(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

> **外键设计决策**：保留外键 + `ON DELETE CASCADE`。理由：sys_user_role 是纯关联表，外键能保证数据一致性（删用户自动删关联，避免孤儿记录）；现有 schema.sql 未禁用外键，风格一致。关联表不加 created_at（无业务意义）。

**表6：sys_refresh_token（refresh token 表，v2.1 自检新增）**

```sql
CREATE TABLE IF NOT EXISTS sys_refresh_token (
    id            INT PRIMARY KEY AUTO_INCREMENT,
    user_id       INT NOT NULL,                          -- 关联用户
    token_hash    VARCHAR(100) NOT NULL,                 -- refresh token 的 SHA-256 哈希（不存明文）
    expires_at    DATETIME NOT NULL,                     -- 过期时间
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES sys_user(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE INDEX idx_refresh_token_user ON sys_refresh_token(user_id);
CREATE INDEX idx_refresh_token_hash ON sys_refresh_token(token_hash);
```

> **为什么需要这张表**：企业级要求"管理员禁用账号后，其 refresh token 立即失效"。不存库的话，禁用账号后用户仍能用 refresh token 换新 access token，安全漏洞。登出时按 user_id 删除该用户所有 refresh token 记录。

**预置数据（首次部署时自动插入）**

```sql
-- 三角色
INSERT INTO sys_role (role_code, role_name, description) VALUES
    ('STAFF', '客服', '录入退货记录，查看自己处理的记录'),
    ('SUPERVISOR', '主管', '审核记录+查看统计+导出数据'),
    ('ADMIN', '管理员', '用户管理+全权限');

-- 默认管理员账号（密码：admin123，BCrypt 哈希，首次登录强制改密）
-- 注意：$2a$10$ 开头是 BCrypt 哈希格式，实际哈希值需用 Spring Security 的 BCryptPasswordEncoder 生成
-- 落地 schema.sql 时用应用启动 ApplicationRunner 检查无 admin 则创建（避免哈希值硬编码不一致）
INSERT INTO sys_user (username, password_hash, display_name, status) VALUES
    ('admin', '$2a$10$PLACEHOLDER_BCRYPT_HASH', '默认管理员', 'active');

INSERT INTO sys_user_role (user_id, role_id)
    SELECT u.id, r.id FROM sys_user u, sys_role r
    WHERE u.username = 'admin' AND r.role_code = 'ADMIN';
```

> **admin 初始化策略**：schema.sql 只建表 + 插入 3 个角色（静态数据）。admin 账号由 ApplicationRunner 在应用启动时检查"无 admin 用户则创建"（用 BCryptPasswordEncoder 生成哈希），避免哈希值硬编码在 SQL 里导致不一致。首次登录强制改密（sys_user 不加 must_change_password 字段，改用"初始密码 admin123 与登录密码相同则强制改"的简单判断，避免过度设计）。

**字段说明**

| 表 | 字段 | 说明 |
|----|------|------|
| sys_user | username | 登录用户名，唯一索引 |
| sys_user | password_hash | BCrypt 哈希，飞书 OAuth 用户可为空（不通过密码登录） |
| sys_user | feishu_user_id | 飞书 user_id，用于 OAuth 绑定，加索引加速查询 |
| sys_user | status | active 可登录，disabled 禁用（禁用后 refresh token 也失效，见 sys_refresh_token） |
| sys_user | last_login_at | 最后登录时间，用于审计 |
| sys_user | created_at / updated_at | 创建/修改时间，所有业务表都应有 |
| sys_role | role_code | STAFF/SUPERVISOR/ADMIN，代码里用此字段判断权限 |
| sys_user_role | (user_id, role_id) | 联合主键，一个用户可有多个角色（预留扩展） |
| sys_refresh_token | token_hash | refresh token 的 SHA-256 哈希，不存明文（防 DB 泄露后 token 被盗用） |
| sys_refresh_token | expires_at | 过期时间，refresh 时校验 |

**OAuth 首次登录策略**

- **不自动创建账号**：用户用飞书 OAuth 登录时，后端用 feishu_user_id 查 sys_user，找不到则返回错误"账号未绑定，请联系管理员绑定"
- **管理员绑定**：管理员在用户管理界面创建用户后，可填入 feishu_user_id 完成绑定
- **理由**：企业级"先审批后访问"原则，避免任何飞书用户授权就能进入系统

**权限矩阵**

| 接口 | STAFF（客服） | SUPERVISOR（主管） | ADMIN（管理员） |
|------|:---:|:---:|:---:|
| `/api/upload/**`（上传识别） | ✅ | ✅ | ✅ |
| `/api/confirm/**`（确认写飞书） | ✅ | ✅ | ✅ |
| `/api/records`（查询记录） | ✅（全部） | ✅（全部） | ✅（全部） |
| `/api/dashboard/**`（仪表盘） | ❌ | ✅ | ✅ |
| `/api/auth/**`（鉴权相关） | ✅ | ✅ | ✅ |
| `/api/admin/**`（用户管理） | ❌ | ❌ | ✅ |

> **客服记录范围决策（重要，避免后续遗忘）**：
> 本期（F01）**客服可见全部记录**，不做按 created_by 过滤。
> 原因：return_records 当前无 created_by 字段，本期不动该表（AGENTS.md 禁止改 DDL 未经讨论）。
> **F03 操作审计时**，会统一给 return_records 加 `created_by` / `updated_by` 字段，届时再做客服记录范围细化（只看自己处理的）。
> 此决策已在 docs/10 第八章备注，F03 落地时必须执行。

**4 表 ER 关系**

```
sys_user (1) ----< (N) sys_user_role >---- (1) sys_role
    │
    └────< (N) sys_refresh_token
```

- 一个用户可有多个角色（sys_user_role 多对多）
- 一个用户可有多个 refresh token（多端登录：PC + 手机）
- 删用户级联删关联和 refresh token（ON DELETE CASCADE）

| user_id | 操作者 user_id，未登录操作（如登录失败）为 NULL |
| username | 冗余存储，避免查询审计时 join sys_user（用户删除后 join 不到） |
| action | 操作类型枚举：LOGIN/LOGOUT/UPLOAD/CONFIRM/DELETE_RECORD/CREATE_USER/UPDATE_USER/DELETE_USER/RESET_PASSWORD/CHANGE_PASSWORD/UPDATE_PROFILE/UPDATE_ROLE |
| target_type | 操作对象类型：return_record/user/auth 等 |
| target_id | 操作对象 ID（如退货记录 ID、用户 ID） |
| description | 操作描述，人类可读 |
| success | 是否成功。失败操作也要审计（如登录失败、权限不足） |
| ip | 操作者 IP，从 HttpServletRequest 取 |
| user_agent | User-Agent，用于追踪操作来源 |
| request_data | 请求参数 JSON，**脱敏后存储**（密码等敏感字段不记录） |
| created_at | 操作时间，加索引支持按时间范围查询 |

**保留期**

- 审计日志**只增不删**（合规要求）
- 保留期 **180 天**（6 个月）
- 超期归档逻辑由 F02 隐私脱敏与保留期管理一并实现

**查询优化**

| 查询场景 | 索引 |
|---------|------|
| 某用户的操作记录 | idx_operation_log_user |
| 某类操作（如所有 LOGIN） | idx_operation_log_action |
| 按时间范围查询 | idx_operation_log_created |
