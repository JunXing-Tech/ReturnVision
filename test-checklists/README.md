# ReturnVision 待测试清单使用规范

> 版本：v1.0 | 日期：2026-07-21
> 作用：每个功能/步骤迭代都对应一份"待测试清单"，从开发前的大纲到开发后的执行记录全流程跟踪，让单人开发的测试有清单可依、有记录可查、有自审可循。

---

## 一、为什么需要清单

单人开发最大的盲区是"改了 A 忘了测 B"。本目录用结构化清单解决：
1. **可追溯**：以后翻看能查到当时测了什么、为什么必须人工、结论是什么。
2. **防遗漏**：固定自审 + 通读两道关把"以为测了但其实没列"的项发现出来。
3. **自动/人工分离**：把"真不能自动化"和"其实可以但偷懒"分开列，前者安心做人工，后者逐步转自动。
4. **回归显式提醒**：减少"改了 A 间接影响到 B"的盲点。

---

## 二、清单跟谁绑定 / 放哪 / 进 git 吗

| 决策 | 结论 |
|------|------|
| 跟谁绑定 | 跟功能/步骤绑定，一份清单跟踪一个功能从开发到完成 |
| 放哪 | `test-checklists/` 独立目录，与 docs 永久方案分离 |
| 写法 | 开发前写大纲 + 开发中勾状态 + 开发后补执行记录 |
| 是否进 git | 进 git，便于以后追溯 |

**文件命名**：`YYYY-MM-DD_功能名.md`，例如 `2026-07-21_F05-字段置信度.md`。

## 三、清单固定结构（6 节，从上到下按顺序填写）

```markdown
# 待测试清单 — [功能名] — [日期]

## 〇、红绿灯标准（本清单完成后到达什么状态可发版）
- P0 项全绿 = 可发版
- P1 项全绿 = 可提交，谨慎发版
- P2 项容许黄/红（标注跳过理由即可）

## 一、基本信息
- 关联文档：docs/xx-xxx.md（验收要点所在）
- 改动范围：Service / Controller / DDL / 配置 / 前端
- git commit：（提交后回填）
- 影响面评估（对照 docs/11 第二章 8 项清单）：逐项打勾或备注"否"

## 二、自动化测试项（可交给 CI / AI 生成）
| 测试ID | 测试目标 | 测试类型 | 优先级 | 状态 | 备注 |
|--------|---------|---------|--------|------|------|

## 三、人工必须验证项（为什么不能自动化必须给理由）
| 验证ID | 验证点 | 人工理由 | 优先级 | 状态 | 负责人 | 验证时间 |
|--------|-------|---------|--------|------|--------|---------|

## 四、回归选择（AI 静态影响分析初稿 + 人工补充）
| 影响模块 | 关联原因 | 应回归测试项 | 已跑 |
|---------|---------|------------|------|

## 五、自审记录（必须按 5.1 → 5.2 → 5.3 → 5.4 顺序）
### 5.1 写完后角色切换通读（开发者视角 → 测试者视角 → 用户视角）
### 5.2 固定检查表（5 项打勾）
### 5.3 AI 辅助反向核对（对照 docs 验收要点 / 权限矩阵）
### 5.4 自审结论：✅ 通过 / ⚠️ 补充了 X 项 / ❌ 重写

## 六、执行记录（事后回填：通过 / 失败 / 跳过 + 失败摘要）
```

## 四、字段填写约定

### 状态标记
- `⬜` 未开始（未生成 / 未执行）
- `🟡` 进行中 / 已生成但未跑通
- `✅` 已通过
- `❌` 失败（必须在执行记录写失败摘要）
- `⏭️` 跳过（必须在备注写跳过理由）

### 优先级
- `P0` = 不测会出大事（核心业务路径 / 数据完整性 / 安全）
- `P1` = 不测可能影响体验但不破坏核心
- `P2` = 边角场景，可延后

### 测试类型
- `单元` = JUnit5 + Mockito，纯逻辑隔离
- `契约` = MockMvc + `@WebMvcTest`，验证 `ResponseResult` 结构
- `集成` = `@SpringBootTest` + Testcontainers / WireMock（后续阶段引入）
- `E2E` = Playwright 驱动前端（后续阶段引入）

### 人工理由（必须填，防止偷懒）
真实可填的例子：
- "依赖真实外部服务（OCR/LLM/飞书），mock 无法体现真实表现"
- "UI 视觉/动效主观感受"
- "HTTPS/部署链路不在自动化覆盖范围"
- **不允许填**："太麻烦"、"没空"、"不确定" — 这种应转为自动化项去努力

## 五、自审顺序（重要，不可颠倒）

### 5.1 角色切换通读（反思式）
写完清单后假想换视角再读一遍。按改动规模分档：

**大改动**（跨模块/新功能/DDL 变更）：必须三轮通读
- **开发者视角**：我清楚自己改了什么，清单是否覆盖我改的所有点？
- **测试者视角**：如果我不是这次改动的人，照这份清单我能不能跑出问题？步骤是否清晰到可执行？
- **用户视角**：用户实际使用时会走的路径，清单是否覆盖？有没有遗漏"看起来不重要但用户会触发"的场景？

**小改动**（单文件 bug fix / 单字段调整）：至少一轮通读，可选视角
- 至少做"测试者视角"通读：照这份清单我能不能跑出问题？
- 若改动会改变用户感知，再加"用户视角"

通读产出："补充的项" + "对原有项的修订"。新发现直接补进二/三/四节对应位置。

### 5.2 固定检查表（校验式，5 项必须全打勾）
- [ ] 对照 docs 验收要点，逐条都有对应测试项？
- [ ] 权限矩阵：3 角色（STAFF/SUPERVISOR/ADMIN）× 涉及接口是否覆盖？
- [ ] DDL 变更是否补了数据迁移测试（如涉及，对照 AGENTS.md"DDL 变更升级验证"）？
- [ ] 外部依赖是否全部 mock / 降级，没有真实调用烧 token / 烧钱？
- [ ] 人工验证项的理由是否充分（非"懒"而是"真不能自动化"）？

### 5.3 AI 辅助反向核对
将 docs 的"验收要点清单"与权限矩阵喂给 AI：
- 让 AI grep 控制器代码，与清单二节比对，输出"覆盖缺口"
- 让 AI grep 权限注解，与清单中权限相关测试项比对
- AI 输出报告：缺口 N 处，已补 M 处，剩余 K 处需人工判断

### 5.4 自审结论
- ✅ 通过：5.2 五项全勾 + 5.3 缺口已处理
- ⚠️ 补充 X 项：5.1 或 5.3 发现遗漏，已补进对应节
- ❌ 重写：清单结构性缺失，回到第一节重做

## 六、可扩展性原则

本目录的清单结构是"最小骨架"。后续可逐步扩展：

| 阶段 | 进阶能力 | 触发时机 |
|------|---------|---------|
| Phase 1（当前） | 目录 + README + 后端测试骨架 | 现已完成 |
| Phase 2 | 清单二节自动生成（AI 读 Service 代码） | ValidatorService 等纯逻辑 Service 测试落地时 |
| Phase 3 | 清单四节 AI 静态影响分析 | `git diff` → grep 调用方脚本成型时 |
| Phase 4 | E2E 清单节（Playwright 场景） | 前端引入 Playwright 时 |
| Phase 5 | 失败日志喂 AI → 自动 bug 报告 | GitHub Actions CI 落地时 |

**新增字段规则**：发现清单缺字段时，先用真实功能跑一次验证是否真有必要，确认后再加入本 README 的"结构"节作为规范升级（v1.1、v1.2...）。不要拍脑袋加字段。

---

## 七、与 AGENTS.md 的衔接

- 本机制不替代 AGENTS.md 的 **L1/L2/L3 分级代码自审**，而是与之配合：
  - 代码自审 → 检查"代码本身"是否正确
  - 测试清单 → 检查"测试是否覆盖"是否正确
- 本机制不替代 docs/00 的 **12 步检查点**，而是给每个检查点提供"测试侧的依据"
- L3 任务的"机制有效性分析报告"应包含对"测试清单使用情况"的回顾
